# SpaceGame — стандарт оформления озарений и первая реализация «Острый край»

## Назначение документа

Этот документ описывает, как в SpaceGame должны работать **озарения** — ранние открытия, которые возникают не через кнопку исследования, а через опыт игрока: сбор ресурсов, наблюдение, повторение действий, выживание и развитие лагеря.

Документ нужен для ИИ-агента в VS Code, чтобы он реализовывал систему озарений единообразно и без самодеятельности.

---

# 1. Главная проблема текущей реализации

В игре уже есть:

- озарения;
- книга знаний;
- исследования.

Но сейчас игрок слабо ощущает, что после озарения что-то реально изменилось. Озарения проходят почти незаметно, новые действия появляются без понятной причинно-следственной связи, а книга знаний воспринимается как отдельная справка, а не как память развития общины.

Нужно изменить подачу.

Озарение должно быть не строкой в логе, а маленьким событием:

```text
Игрок что-то сделал
↓
Персонаж что-то заметил
↓
Появляется визуальная сцена
↓
Игрок участвует в коротком наблюдении/выборе
↓
Происходит озарение
↓
Игра показывает, что изменилось
↓
В книгу знаний добавляется запись
↓
Открывается или приближается новая возможность
```

---

# 2. Что такое озарение в SpaceGame

## 2.1. Озарение — это не исследование

Озарение не должно запускаться кнопкой «изучить».

Озарение возникает из практики.

Примеры:

- игрок несколько раз собирает камни;
- игрок замечает камень с острым сколом;
- появляется озарение «Острый край».

Или:

- игрок собирает ветки;
- замечает, что не все ветки одинаково ломаются;
- появляется озарение «Прочная палка».

Или:

- игрок собирает волокна;
- понимает, что ими можно связывать предметы;
- появляется озарение «Связка волокон».

## 2.2. Исследования — это другой тип прогресса

Исследования должны появляться позже, когда у игрока уже есть:

- лагерь;
- время;
- ресурсы;
- повторяемые действия;
- зачатки организации.

Исследование — это осознанный проект.

Озарение — это наблюдение, пришедшее из опыта.

---

# 3. Общий принцип оформления всех озарений

Каждое озарение должно состоять из следующих частей.

## 3.1. Условие появления

У каждого озарения должно быть конкретное условие.

Примеры:

```js
(game.resourceTotals.stone || 0) >= 4;
```

```js
(game.resourceTotals.wood || 0) >= 4;
```

```js
(game.resourceTotals.fiber || 0) >= 3;
```

Важно использовать **resourceTotals**, если озарение связано с накопленным опытом сбора, а не с текущим количеством ресурса.

Например:

- `resourceTotals.stone` — сколько камней игрок всего собрал;
- `resources.stone` — сколько камней сейчас лежит в запасе.

Для озарений чаще нужен именно `resourceTotals`.

## 3.2. Смысл открытия

Каждое озарение должно отвечать на вопрос:

```text
Что персонаж понял такого, чего раньше не понимал?
```

Не просто:

```text
Открыт новый рецепт.
```

А:

```text
Персонаж понял, что форма камня может быть важнее его веса.
```

## 3.3. Визуальная discovery-сцена

Ключевые озарения должны показываться через отдельную визуальную сцену.

Сцена должна содержать:

- изображение;
- заголовок;
- короткий атмосферный текст;
- простое участие игрока;
- результат;
- блок «Что изменилось».

## 3.4. Простое участие игрока

Где уместно, игрок должен сделать маленькое действие:

- выбрать правильный предмет;
- выбрать верное наблюдение;
- подтвердить понимание;
- позже можно добавить простое соединение предметов.

Важно:

- это не должна быть сложная мини-игра;
- интерактив должен занимать 5–15 секунд;
- ошибочный выбор не должен ломать прогресс;
- при ошибке нужно дать мягкую подсказку.

## 3.5. Блок «Что изменилось»

После озарения обязательно показать игроку, что изменилось.

Пример:

```text
Что изменилось:

✅ Открыто знание: Острый камень
✅ Камни теперь воспринимаются не только как ресурс, но и как материал для инструментов
✅ Появился первый шаг к созданию грубого орудия
➡ Следующий шаг: найти прочную палку и волокна
```

## 3.6. Запись в книгу знаний

Каждое важное озарение должно добавлять запись в книгу знаний.

Книга знаний должна объяснять:

- как было открыто знание;
- что изменилось после открытия;
- с какими знаниями оно связано;
- к чему оно ведёт.

# 4. Единая структура данных для озарений

Для каждого озарения использовать общий формат.

Примерная структура:

```js
insight_id: {
  id: "insight_id",
  order: 1,
  name: "Название озарения",
  icon: "🧠",

  conditionText: "Открывается после ...",
  condition: (game) => true,

  description:
    "Краткое описание смысла озарения.",

  momentText:
    "Что персонаж заметил в момент открытия.",

  unlockText:
    "Что стало понятно после открытия.",

  outcomes: [
    "Что изменилось 1",
    "Что изменилось 2",
    "Что изменилось 3"
  ],

  knowledgeEntry: "knowledge_entry_id",

  discoveryScene: {
    id: "insight_id_scene",
    insightId: "insight_id",
    title: "Название озарения",
    image: "assets/insights/insight_id.webp",
    alt: "Описание изображения",

    introText:
      "Короткий текст перед интерактивом.",

    observationText:
      "Что именно персонаж замечает.",

    interaction: {
      type: "choice",
      prompt: "Вопрос игроку",
      options: [
        {
          id: "wrong_option",
          label: "Неверный вариант",
          result: "Мягкая подсказка."
        },
        {
          id: "correct_option",
          label: "Правильный вариант",
          correct: true,
          result: "Подтверждение правильного наблюдения."
        }
      ]
    },

    successTitle: "Короткий вывод",
    successText:
      "Главный смысл озарения после правильного выбора.",

    effectsTitle: "Что изменилось",
    effects: [
      {
        type: "knowledge_entry",
        target: "knowledge_entry_id",
        label: "Открыто знание: ..."
      },
      {
        type: "insight_progress",
        target: "some_future_unlock",
        label: "Прогресс к ..."
      },
      {
        type: "next_step",
        target: "next_insight_or_action",
        label: "Следующий шаг: ..."
      }
    ]
  }
}
```

# 5. Типы интерактива

На первом этапе реализовать минимум два типа.

## 5.1. `choice`

Игрок выбирает один вариант из нескольких.

Если выбран неверный вариант:

- показать `result`;
- не закрывать сцену;
- дать выбрать снова.

Если выбран правильный вариант:

- показать `result`;
- показать `successText`;
- показать блок «Что изменилось»;
- разрешить закрыть сцену.

## 5.2. `confirm`

Сцена без выбора.

Используется там, где выбор не нужен, например:

- первый костёр;
- завершение важного перехода;
- крупное событие лагеря.

# 6. Состояние игры для discovery-сцен

Добавить безопасные поля состояния:

```js
this.discoveryEvents = [];
this.pendingDiscoveryScene = null;
this.seenDiscoveryScenes = {};
```

## 6.1. Назначение

### `discoveryEvents`

История событий открытия.

### `pendingDiscoveryScene`

Сцена, которую нужно показать игроку.

### `seenDiscoveryScenes`

Список уже показанных сцен, чтобы одна и та же сцена не всплывала повторно.

## 6.2. Совместимость сохранений

Если в старом сохранении этих полей нет, игра должна создать значения по умолчанию.

Старые сохранения не должны ломаться.


- крупную иконку озарения;
- декоративный блок;
- alt-текст не показывать как обычный текст, но использовать для доступности.

# 8. Принцип цепочки ранних озарений

Ранние озарения должны складываться в логическую технологическую цепочку.

Пример:

```text
Острый край
↓
Прочная палка
↓
Связка волокон
↓
Материалы работают вместе
↓
Грубое орудие
```

Важно:

**Озарение «Острый край» не должно сразу открывать готовое грубое орудие.**

Оно открывает только первый шаг:

```text
Камень может быть режущим материалом.
```

Грубое орудие должно стать доступно только после нескольких связанных наблюдений.

# 9. Первая реализация: озарение «Острый край»

## 9.1. Когда появляется

Озарение появляется после того, как игрок суммарно собрал 4 единицы ресурса `stone`.

Условие:

```js
(game.resourceTotals.stone || 0) >= 4;
```

Использовать именно `resourceTotals.stone`.

## 9.2. Почему появляется

Игрок несколько раз подбирает камни. Среди обычных округлых камней персонаж замечает один камень со сколом.

Он не самый большой и не самый тяжёлый, но его край отличается.

Персонаж берёт его в руку и понимает:

- край цепляет кожу;
- край оставляет след на сухой ветке;
- такой камень может пригодиться;
- камень может быть не просто грузом, а инструментальным материалом.

## 9.3. Главный смысл

```text
Форма важнее веса.
Острый скол может резать.
Камень можно выбирать не только по количеству, но и по свойствам.
```

Но предпочтительно WEBP.

# 12. Текст модального окна для игрока

## Заголовок

```text
🧠 Озарение: Острый край
```

## Основной текст

```text
Среди собранных камней один оказался другим.

Он не был самым большим.
Не был самым тяжёлым.
Но его край цеплялся за кожу, оставлял след на сухой ветке и будто сам подсказывал руке, что с ним можно сделать.

До этого камень был просто камнем — грузом, опорой, тем, что можно бросить или положить в круг очага.

Теперь стало ясно: форма важнее веса.
Острый скол может резать.
```

## Вопрос игроку

```text
Что делает этот камень особенным?
```

## Варианты ответа

```text
Он тяжелее остальных
Он более круглый и удобный
У него острый скол
```

## Реакция на неверный выбор

Для варианта «Он тяжелее остальных»:

```text
Вес может пригодиться, но сейчас важнее не тяжесть. Внимание тянется к краю.
```

Для варианта «Он более круглый и удобный»:

```text
Круглый камень удобно держать, но он не объясняет, почему ветка царапается.
```

## Реакция на правильный выбор

```text
Да. Скол цепляет кожу и оставляет след на сухой ветке. Этот камень может резать.
```

## Вывод

```text
Камень может быть не только грузом.
Он может стать продолжением руки.
```

## Блок «Что изменилось»

```text
✅ Открыто знание: Острый камень
✅ Камни теперь воспринимаются не только как ресурс, но и как материал для инструментов
✅ Появился первый шаг к созданию грубого орудия
➡ Следующий шаг: найти прочную палку и волокна
```

# 13. Запись в книгу знаний `sharp_stone`

Обновить или добавить запись:

```js
sharp_stone: {
  id: "sharp_stone",
  order: 1,
  title: "Острый камень",

  discoveredBy: "Озарение: Острый край",

  lines: [
    "Не каждый камень одинаково полезен. Размер и вес важны не всегда: иногда главное — форма края.",
    "Острый скол может резать, скоблить и оставлять след на дереве, коже или волокнах.",
    "Так появляется первое понимание: материал можно выбирать не только по количеству, но и по свойствам."
  ],

  changed: [
    "Камень начинает восприниматься как возможный инструмент",
    "Открывается первый шаг к созданию грубого орудия"
  ],

  related: [
    "Прочная палка",
    "Связка волокон",
    "Материалы работают вместе"
  ]
}
```

Если текущий UI книги знаний не поддерживает поля `discoveredBy`, `changed`, `related`, добавить поддержку аккуратно.

Если это слишком большой объём для первого шага, разрешается:

1. сохранить поля в данных;
2. пока отрисовать только `lines`;
3. позже доработать визуальное оформление книги знаний отдельной задачей.

# 16. Как должна работать сцена в игре

Полный сценарий:

1. Игрок начинает новую игру.
2. Игрок собирает камни.
3. После 1-го камня озарения нет.
4. После 2-го камня озарения нет.
5. После 3-го камня озарения нет.
6. После 4-го суммарно собранного камня срабатывает `sharp_edge`.
7. Игра открывает discovery-сцену «Острый край».
8. В сцене показывается изображение `sharp_edge.webp`.
9. Игрок видит текст наблюдения.
10. Игрок отвечает на вопрос: «Что делает этот камень особенным?»
11. Если игрок ошибся — показывается мягкая подсказка.
12. Если игрок выбрал «У него острый скол» — показывается вывод.
13. Показывается блок «Что изменилось».
14. Игрок закрывает сцену.
15. В лог добавляется событие.
16. В книгу знаний добавляется запись «Острый камень».
17. Сцена больше не появляется повторно.
18. После перезагрузки страницы сцена не должна открываться снова.

# 17. Acceptance Criteria

## Общие критерии системы

1. Озарения оформляются по единому принципу:
   - условие;
   - момент наблюдения;
   - визуальная сцена;
   - участие игрока;
   - результат;
   - блок «Что изменилось»;
   - запись в книгу знаний.

2. Озарения не являются исследованиями по кнопке.

3. Ошибочный выбор в discovery-сцене не ломает прогресс.

4. Старые сохранения не ломаются.

5. Код остаётся модульным.

6. Изображения загружаются из `web/assets/insights/`.

7. Если изображения нет, игра показывает fallback.

## Критерии для `sharp_edge`

1. `sharp_edge` появляется после суммарного сбора 4 камней.
2. Используется `resourceTotals.stone`, а не `resources.stone`.
3. При первом открытии показывается discovery-сцена.
4. В сцене отображается изображение `assets/insights/sharp_edge.webp`.
5. Сцена содержит вопрос: `Что делает этот камень особенным?`
6. Есть минимум три варианта ответа.
7. Правильный вариант: `У него острый скол`.
8. Неверные варианты дают мягкую подсказку.
9. После правильного выбора показывается блок «Что изменилось».
10. Добавляется запись `sharp_stone` в книгу знаний.
11. В лог добавляется понятная запись.
12. Сцена не появляется повторно после закрытия.
13. После перезагрузки сохранённой игры сцена не появляется повторно.
14. Озарение не открывает сразу готовое грубое орудие.
15. Озарение только приближает создание грубого орудия.

# 18. Что не делать в этой задаче

Не реализовывать сразу все озарения.

Не делать полноценное древо технологий.

Не менять баланс всех ресурсов.

Не переписывать весь UI исследований.

Не удалять существующие тексты без необходимости.

Не делать сложные мини-игры.

Не использовать современные предметы, металл, магию или фантастические эффекты в визуалах.

# 20. Будущие озарения должны оформляться по этому же шаблону

После успешной реализации «Острого края» остальные озарения оформлять по тому же принципу:

```text
Условие появления
↓
Что персонаж заметил
↓
Визуальная сцена
↓
Короткое участие игрока
↓
Что стало понятно
↓
Что изменилось
↓
Запись в книгу знаний
↓
Следующий шаг
```

Но условия появления у каждого озарения будут свои.

Примеры будущих условий:

```js
// Прочная палка
(game.resourceTotals.wood || 0) >=
  4(
    // Связка волокон
    game.resourceTotals.fiber || 0,
  ) >=
  3;

// Материалы работают вместе
game.insights.sharp_edge &&
  game.insights.sturdy_branch &&
  game.insights.fiber_bindings &&
  (game.resources.wood || 0) >= 2 &&
  (game.resources.stone || 0) >= 2 &&
  (game.resources.fiber || 0) >=
    2(
      // Сухая растопка
      game.resourceTotals.wood || 0,
    ) >=
    6 &&
  (game.resourceTotals.fiber || 0) >= 3 &&
  game.insights.sturdy_branch &&
  game.insights.fiber_bindings;
```

Важно: не реализовывать эти будущие сцены в текущей задаче, если пользователь прямо не попросил. Сейчас задача — сделать качественный первый эталон на примере **«Острого края»**.

# 22. Общий стандарт для всех изображений озарений

Изображение должно показывать не общий пейзаж и не красивый постер, а **момент наблюдения**, из которого рождается понимание.

Правильный принцип кадра:

```text
рука / взгляд / материал / след действия
```

То есть игрок должен за 1–2 секунды понять:

- что именно персонаж заметил;
- какой предмет или материал стал важным;
- почему это озарение отличается от предыдущего.

## 22.1. Что должно быть видно почти всегда

- один главный предмет в фокусе;
- рука, жест или поза персонажа, показывающая внимание к предмету;
- след взаимодействия: царапина, связка, тление, круг из камней, запас у очага;
- фон раннего мира без лишних деталей;
- освещение, которое подчёркивает предмет, а не пейзаж.

## 22.2. Что лучше избегать

- дальних общих планов;
- нескольких людей в кадре;
- современных предметов, одежды и форм;
- слишком чистого или героического фэнтези-вида;
- перегруженного фона;
- слишком тёмных сцен, где главный объект теряется при уменьшении.

## 22.3. Практический формат

- предпочтительный формат исходника: вертикальный 4:5 или 3:4;
- рабочий минимум по размеру: 1200x1500 px;
- главный объект должен занимать примерно 35–55% кадра;
- лицо, рука и ключевой предмет лучше держать в центральной безопасной зоне шириной примерно 60% и высотой примерно 65% кадра;
- не ставить главный предмет вплотную к краям: миниатюра схемы озарений делает жёсткий кроп;
- кадр должен оставаться читаемым в трёх режимах: большая discovery-сцена, средняя карточка выбранного озарения и узкая миниатюра в схеме.

## 22.4. Использование в UI

Сейчас одно и то же изображение может использоваться:

- в discovery-сцене;
- в правой карточке выбранного озарения;
- в маленькой карточке узла на карте озарений.

Поэтому картинка должна работать и как большая атмосферная сцена, и как компактный визуальный маркер.

Практически это значит следующее:

- в схеме озарений изображение показывается как узкая портретная миниатюра;
- в правой карточке тоже используется более плотный портретный кроп;
- для очень сложных кадров может понадобиться отдельная настройка позиции кадра в данных озарения;
- если важный объект читается только на дальнем плане, такой кадр для схемы не подойдёт.

# 23. Арт-брифы по текущим озарениям

## 23.1. `sharp_edge.webp` — «Острый край»

Это уже хороший эталон.

Почему он удачный:

- есть лицо и реакция персонажа;
- есть рука с предметом;
- острый скол читается сразу;
- рядом лежат обычные камни для контраста;
- сцена показывает именно момент понимания, а не общий быт.

Его стоит использовать как образец для остальных кадров.

## 23.2. `sturdy_branch.webp` — «Прочная палка»

Что должно происходить:

- персонаж перебирает ветви;
- одна палка не ломается как остальные или естественно ложится в руку;
- видно, что это не хворост для костра, а заготовка для основы орудия.

Что желательно показать в кадре:

- рука с крепкой прямой палкой;
- рядом более кривые, ломкие или сухие ветви;
- след проверки на прочность: изгиб, упор в землю, надлом у других веток.

Главный смысл кадра:

```text
не любая ветка одинакова — одна из них годится как основа
```

## 23.3. `fiber_bindings.webp` — «Связка волокон»

Что должно происходить:

- персонаж скручивает волокна пальцами;
- видно, что по отдельности это слабые нити, а вместе они уже держат форму.

Что желательно показать в кадре:

- крупный план рук и волокон;
- начало простой скрутки или узла;
- рядом россыпь несвязанных волокон для сравнения.

Главный смысл кадра:

```text
слабое по отдельности становится полезным вместе
```

## 23.4. `materials_work_together.webp` — «Материалы работают вместе»

Что должно происходить:

- персонаж соединяет каменный скол, палку и волокна;
- это ещё не готовый красивый инструмент, а момент первой сборки.

Что желательно показать в кадре:

- три материала одновременно в кадре;
- руки, которые удерживают их вместе;
- ощущение первой грубой сборки, а не завершённого ремесленного изделия.

Главный смысл кадра:

```text
полезность рождается из соединения, а не из одной находки
```

## 23.5. `dry_tinder.webp` — «Сухая растопка»

Что должно происходить:

- персонаж отделяет сухую мелочь от грубых веток;
- видно, что именно тонкая сухая смесь лучше ловит жар.

Что желательно показать в кадре:

- сухая трава, тонкая кора, ломкие веточки, волокно;
- рядом более толстая сырая ветвь, которая выглядит менее подходящей;
- можно показать слабый дымок или первые признаки схватывания жара, но не большой костёр.

Главный смысл кадра:

```text
огонь начинается с малого и сухого, а не с большого полена
```

## 23.6. `hearth_circle.webp` — «Круг очага»

Что должно происходить:

- персонаж выкладывает или рассматривает круг из камней вокруг выбранного места для огня;
- место начинает читаться как очаг, а не случайный костёр на земле.

Что желательно показать в кадре:

- круг или полукруг камней;
- расчищенную землю;
- маленький удержанный жар или подготовленное место под него;
- руки, которые укладывают камни или поправляют их.

Главный смысл кадра:

```text
огонь живёт дольше там, где место для него выбрано и оформлено
```

## 23.7. `hearth_needs_reserve.webp` — «Очаг любит запас»

Что должно происходить:

- рядом с очагом уже виден заранее собранный запас топлива и растопки;
- смысл не в самом пламени, а в подготовленности.

Что желательно показать в кадре:

- очаг на втором плане;
- аккуратный запас сухих ветвей, травы, волокон в пределах досягаемости;
- персонаж, который складывает запас рядом, а не ищет его в спешке.

Главный смысл кадра:

```text
очаг держится не на искре, а на заранее подготовленном запасе
```
