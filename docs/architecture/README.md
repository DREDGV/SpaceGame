# SpaceGame Architecture Map

Эта папка содержит расширяемую визуальную архитектурную карту SpaceGame. Карта устроена как data-driven инструмент: данные отдельно, стили отдельно, HTML-просмотрщик отдельно.

**Полные правила проектирования и работы с картой:** [`ARCHITECTURE_MAP_RULES.md`](ARCHITECTURE_MAP_RULES.md) — эпохи, статусы, запреты на изменение прототипа, чеклисты и стратегическая последовательность.

## Как открыть карту

Открыть файл в браузере:

```text
docs/architecture/SPACEGAME_ARCHITECTURE_MAP.html
```

## Как пользоваться интерактивной картой

- Используйте **поиск** для быстрого нахождения эпох, главных систем, матрицы, зависимостей и экранов.
- Фильтр **«Все веса»** / лёгкий–ядро — по полю `designWeight` эпохи (проектная метка, не баланс прототипа).
- Используйте **верхнюю линию эпох** (карточки A–K) для перехода к эпохе; при необходимости скопируйте ссылку с hash (`#era-B`).
- Используйте **A–C / D–F / G–K** для просмотра групп эпох (остальные карточки скрываются; линия эпох и матрица следуют фильтру).
- Режимы **Обзор** / **Системы** / **Детально** в шапке: обзор компактнее, «Системы» выделяют центральную магистраль, «Детально» — полное раскрытие блоков эпох внизу (как раньше «Подробно»).
- Кнопки **«Свернуть периоды»** / **«Развернуть периоды»** управляют раскрытием секции «Периоды» во всех видимых эпохах.
- Новые данные добавляйте в **`spacegame-architecture-data.js`**; HTML и CSS меняйте только при изменении просмотрщика.

## Как пользоваться dashboard-картой

1. Выберите эпоху в **верхней линии** (или откройте страницу с `#era-*` в URL) — страница прокрутит к карточке эпохи на шкале.
2. Выберите систему в блоке **«Магистраль систем и связей»** — включится фокус: подсветка связей и панель справа.
3. Смотрите **связи** и поля системы в правой панели; связанные системы можно переключать кликом; **Сбросить фокус** или повторный клик по активной карточке снимает выделение.
4. Используйте **поиск** и фильтры эпох/статуса/веса — подсветка совпадений и согласованность линии эпох, матрицы и шкалы.
5. Переходите к **сквозным системам** через правый список или якорь «Сквозные системы» слева для подробного анализа A–K.

```js
window.SPACEGAME_ARCHITECTURE = {
```

## Главный системный каркас (`systems[]`)

Канонический набор из **20** систем (`time`, `people`, `needs`, `labor`, `resources`, `storage`, `buildings`, `production`, `knowledge`, `research`, `logistics`, `trade`, `economy`, `governance`, `security`, `events`, `map`, `goals`, `automation`, `ui`) задаётся в **`systems[]`** вместе с полями **`summary`**, **`roleInGame`**, **`category`** (`spine` | `core`), **`relatedSystems`** и эпохами **`appearsIn`** / **`becomesCoreIn`**. В просмотрщике — секция **«Главные системы игры»** (`#section-core-systems`). Дополнительные системы (например `craft`, `energy`) остаются в том же массиве для матрицы и зависимостей. Подробнее — [`ARCHITECTURE_MAP_RULES.md`](ARCHITECTURE_MAP_RULES.md) (раздел «Главный системный каркас») и [`ARCHITECTURE_SCHEMA.md`](ARCHITECTURE_SCHEMA.md).

### Как добавлять новую идею

1. Определить **главную систему** (или несколько) из канонического каркаса.
2. Определить **эпоху** и смысл в `eras[]`.
3. При необходимости — связь в **`dependencies[]`** или пункт **`roadmap[]`**.
4. Только затем — новая строка в **каталоге игры** (`gameResources`, …), если нужна именованная сущность.

## Сквозные системы (`systemProgression`)

Добавляйте или правьте объекты в массиве **`systemProgression`** в `spacegame-architecture-data.js`: поле **`systemId`** должно совпадать с **`systems[].id`**, поле **`byEra`** — только ключи **A–K** со строками по смыслу эпохи; опционально **`designNotes`** (массив строк); опционально **`relatedSystems`** (массив `{ systemId, relation, note }` — id цели должен быть в **`systems[]`**). В просмотрщике откройте секцию **«4. Сквозные системы»** (`#section-system-progression`): таблица строк реагирует на фильтр эпох A–C / D–F / G–K и на поиск; есть компактная навигация между карточками и блок связанных систем.

### Как добавлять новую сквозную систему

1. Добавить объект в **`systemProgression`** (уникальный **`systemId`** из **`systems[]`**).
2. Заполнить **`byEra`** для всех эпох **A–K** (пока валидатор жёстко требует полный набор для **`people`** и **`labor`**; для новых id правило можно расширить в `validate-architecture-map.mjs`).
3. Добавить **`relatedSystems`** там, где зависимости очевидны.
4. Проверить секцию в **`SPACEGAME_ARCHITECTURE_MAP.html`** (навигация, ссылки, «ещё не описано»).
5. Запустить **`node tools/validate-architecture-map.mjs`**.

## Проектные поля эпохи (вес, роль, плотность)

В каждом объекте `eras[]` заданы поля **`role`**, **`designWeight`**, **`gameplayDensity`**, **`playerTimeShare`**, **`contentGuideline`**. Это **не** игровые характеристики прототипа, а метки для дизайна: куда вкладывать глубину, какую долю времени игрока ожидать, как не раздувать эпоху.

**Как пользоваться при планировании:** перед добавлением большого блока целей, событий или строк каталога свериться с `designWeight` и `contentGuideline` эпохи; для «лёгких» эпох не раздувать списки «для симметрии»; для **core**-эпох допускается больше систем и каталога, но всё равно с привязкой к системам и эпохам. Критерии перехода к следующей эпохе фиксируйте в **`transition.checklist`** (короткие проверяемые пункты, не лор). Подробнее — [`ARCHITECTURE_MAP_RULES.md`](ARCHITECTURE_MAP_RULES.md) (разделы «Вес, роль и плотность эпох» и «Переходы как чеклисты») и [`ARCHITECTURE_SCHEMA.md`](ARCHITECTURE_SCHEMA.md).

Проверка структуры после правок:

```bash
node tools/validate-architecture-map.mjs
```

## Что не трогать без необходимости

- HTML-структуру, если меняются только данные.
- CSS, если меняются только эпохи, периоды, системы, зависимости или roadmap.
- Игровые файлы в `web/prototype`.

## Как добавить эпоху

Добавить объект в массив `eras`.

Минимально нужны:

- `id`;
- `title`;
- `dates`;
- `summary`;
- `timePace`;
- `status`;
- `role`, `designWeight`, `gameplayDensity`, `playerTimeShare`, `contentGuideline`;
- `periods`;
- `systems`;
- `transition` (в т.ч. непустой **`transition.checklist`** для осмысленного перехода к `to`).

## Как добавить критерии перехода

В объекте `era.transition` добавьте массив **`checklist`** из коротких строк (3–7 пунктов): что должно быть в дизайне или в реализации, чтобы переход к эпохе `to` был оправдан. См. `ARCHITECTURE_MAP_RULES.md` — «Переходы как чеклисты».

## Как добавить период

Добавить объект в `era.periods`:

```js
{
  id: "A3",
  title: "Название периода",
  dates: "...",
  summary: "..."
}
```

## Как добавить систему

Добавить объект в `systems`:

```js
{
  id: "systemId",
  title: "Название",
  description: "Что делает система",
  appearsIn: "A",
  becomesCoreIn: "C",
  status: "designing",
  notes: "Проектные заметки"
}
```

## Как добавить зависимость

Добавить объект в `dependencies`:

```js
{
  from: "people",
  to: "needs",
  type: "requires",
  description: "Люди требуют еды, воды, жилья и безопасности"
}
```

## Статусы

- `prototype` — уже частично есть в прототипе.
- `designing` — проектируется сейчас.
- `planned` — запланировано для будущих этапов.
- `later` — поздняя игра или отложенный слой.
- `needs_rework` — требует пересмотра (см. просмотрщик и легенду).
- `blocked` — заблокировано зависимостями до появления условий.

Подробнее о смысле статусов и работе с эпохами — в [`ARCHITECTURE_MAP_RULES.md`](ARCHITECTURE_MAP_RULES.md).

## Правило развития

Если новая идея не помещается в эпоху, систему, зависимость или переход, сначала расширяется карта. Игровой код меняется только после того, как место идеи в архитектуре понятно.
