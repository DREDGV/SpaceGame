# SpaceGame — правила работы с архитектурной картой игры

Главная задача проекта на этом этапе — не наращивать новые игровые механики в прототипе, а выстроить **архитектурную карту игры**: визуальную, расширяемую, data-driven карту, по которой дальше проектируется вся игра.

## 1. Что мы делаем

Создаётся **рабочий архитектурный инструмент**, а не только документ и не только статичная HTML-страница. Он задаёт проектирование сверху вниз:

**время → эпохи → периоды → системы → зависимости → цели → события → ограничения → переходы → реализация.**

Идея игры: SpaceGame — экономическая стратегия развития цивилизации от позднего выживания (порядка **~10 000 до н.э.**) до планетарной цивилизации, космической экономики и дальнего космоса.

Путь игрока:

**один персонаж → стоянка → ранняя община → поселение → города → государства → индустрия → современная технологическая цивилизация → планетарная экономика → космос → дальние колонии.**

## 2. Главный принцип проектирования

Не проектируем игру от отдельных механик («кнопка», «здание», «ресурс»). Порядок мыслей:

1. Какая эпоха?
2. Какой период внутри эпохи?
3. Какой образ жизни?
4. Какие люди?
5. Какой труд?
6. Какие ресурсы?
7. Какие постройки?
8. Какие знания?
9. Какие ограничения?
10. Какие цели?
11. Какие события?
12. Какой переход к следующей эпохе?

Только после этого — реализация в коде.

## 2a. Dashboard-слой карты

**Визуальная карта должна помогать навигации и проектированию, а не просто увеличивать количество блоков.**

- **Dashboard-слой** (шапка, линия эпох, боковая навигация, магистраль систем с фокусом, правая колонка) — это **обзор и управление вниманием**: куда смотреть дальше и какие связи важны сейчас.
- **Подробные таблицы, матрица, каталоги и длинные списки** остаются **ниже** в блоке **«Подробные данные карты»** (сворачиваемый раздел): они не заменяются декоративными панелями, а дополняются ими.
- **Сопровождение просмотрщика:** `SPACEGAME_ARCHITECTURE_MAP.html` — только разметка; `spacegame-architecture-map.js` — рендер и интерактив; `spacegame-architecture-map.css` — стили. Данные — только в `spacegame-architecture-data.js`.
- **Focus mode** по системе в магистрали — основной способ изучать `relatedSystems`; после правок data запускать `node tools/validate-architecture-map.mjs`.

## 3. Главный закон времени

Старт исторической шкалы — **примерно ~10 000 до н.э.** Не уходить в миллионы лет и глубокую предысторию.

**Ранние эпохи проходят быстрее, поздние — медленнее, плотнее и дороже по усилиям** (аналогия: добыча Bitcoin — сначала частый прогресс, затем каждый шаг тяжелее).

Ранние эпохи быстро дают ощущение движения: ручной труд → огонь → стоянка → распорядок → малая группа → поселение. Поздняя игра может быть почти бесконечной: индустрия → современность → планета → космос → дальние колонии.

## 6. Что нельзя делать без прямой команды

- Не трогать игровой код: `web/prototype/`, `game-state.js`, `app.js`, save/load, `SAVE_VERSION`, механики прототипа (очереди, крафт, сбор, экономическое ядро и т.д.).
- Не превращать карту в React/Vue-приложение.
- Не подключать внешние CDN и тяжёлые зависимости.

Карта живёт отдельно от прототипа, пока архитектура не зафиксирована.

## 7. Как устроена карта (data-driven)

```js
window.SPACEGAME_ARCHITECTURE = {
  meta: {},
  eras: [],
  systems: [],
  systemProgression: [],
  dependencies: [],
  roadmap: [],
  uiScreens: [],
  gameCatalog: {},
  gameResources: [],
  gameResearch: [],
  gameTechnologies: [],
  gameEnterprises: []
};
```

HTML — просмотрщик. CSS — внешний вид. Data.js — источник правды.

## Сквозные системы

Массив **`systemProgression`** — отдельный слой от матрицы «эпоха × система»: для выбранных систем задаётся **одна связная история по A–K** (роль системы в каждой эпохе, демография, институты и т.д.). В **`eras[].systems.<id>`** остаются короткие строки для карточки эпохи и быстрого чтения; в **`systemProgression`** — развёрнутый сквозной текст, заметки дизайна и при необходимости **`relatedSystems`** (рабочие связи с другими системами для навигации и понимания архитектуры, а не «декор»). Фильтр эпох A–C / D–F / G–K в просмотрщике сужает **строки таблицы** сквозной системы, не удаляя саму карточку. Редактировать: `spacegame-architecture-data.js` → `systemProgression`; схема полей — `ARCHITECTURE_SCHEMA.md`.

Правила ведения слоя:

1. Новые сквозные системы добавляются **по одной**, с осмысленным `summary` и полным **`byEra` A–K** (для зарегистрированных в валидаторе систем — см. `ARCHITECTURE_SCHEMA.md`).
2. Каждая система с очевидными зависимостями должна иметь **`relatedSystems`**: ссылки на id из **`systems[]`**, краткий **`relation`**, поясняющий **`note`**.
3. Связи нужны для **навигации и архитектуры**: карта не должна превращаться в список таблиц; новое описание связывается с эпохами и с другими системами.
4. Если у связанного **`systemId`** ещё нет своей карточки в `systemProgression`, в HTML показывается «ещё не описано» — это нормальный этап проектирования.
5. После правок — `node tools/validate-architecture-map.mjs` и проверка секции «Сквозные системы» в HTML.

## Главный системный каркас

Массив **`systems[]`** задаёт **крупные механические области** игры (см. раздел «Главные системы игры» в HTML): не энциклопедия, а **каркас**, к которому привязываются эпохи, зависимости, цели и каталог сущностей. Длинные эссе не хранить в `systems[]` — для развития одной системы по эпохам использовать **`systemProgression[]`**, для конкретных ресурсов и технологий — **`gameResources`** / **`gameTechnologies`** / и т.д.

Правило потока идей:

1. Новая идея сначала привязывается к **одной или нескольким главным системам** из канонического набора из 20 id.
2. Затем — к **эпохе** и текстам в **`eras[]`**.
3. Затем — к **`dependencies[]`** и задачам roadmap.
4. Только после этого — новые строки в **каталоге игры**, если нужна именованная сущность.

**Кликабельность:** каждая новая сущность в data должна быть доступна через **инспектор** (клик по карточке в каталоге, системе, эпохе или явная ссылка `#entity:kind:id`). Список ради списка не добавлять.

**Каталог:** у записи должны быть осмысленные **`eraFrom` / `eraTo`**, при возможности **`systemIds`** и связи (`requires`, `unlocks`, `usedIn`, …); иначе объект не помогает проектированию.

### Паспорта сущностей каталога

- Новая сущность каталога должна иметь минимум: `id`, `title`, `summary`, `status`, `eraFrom`, `eraTo`, `systemIds`.
- Каталог не должен быть списком красивых названий: каждая запись должна объяснять, зачем она нужна в экономике, эпохе и системе.
- Если сущность нельзя связать с эпохой и системой, она пока остаётся идеей, а не элементом каталога.
- Полный паспорт добавляется постепенно: `availabilityCondition`, `requires`, `unlocks`, `usedIn`, `producedBy`, `storedIn`, `transportedBy`, `riskNotes`, `prototypeRefs`.
- Пустые секции не добавлять ради симметрии. Если данных пока нет, лучше оставить поле отсутствующим и вернуться к нему после стабилизации эпохи и системы.

Опорная формула (смысловой скелет, не жёсткий геймплейный pipeline):

Время задаёт эпоху.  
Люди создают потребности и труд.  
Труд добывает ресурсы.  
Ресурсы идут в постройки и производство.  
Постройки организуют жизнь и экономику.  
Производство создаёт товары, инструменты и инфраструктуру.  
Знания открывают новые способы действия.  
Логистика связывает территорию.  
Торговля превращает излишки в экономику.  
Управление удерживает масштаб.  
Риски проверяют устойчивость.  
Цели ведут игрока дальше.

## 8. Целевая структура эпохи в `data.js`

Каждая эпоха **стремится** к полной структуре (пустые массивы и отсутствующие ключи допустимы; секции в HTML не показываются, если данных нет):

- `id`, `title`, `dates`, `summary`, `timePace`, `status`
- `periods[]` — `{ id, title, dates, summary }`
- `goals[]`, `events[]`, `discoveries[]`, `limits[]`
- `systems` — объект с текстовыми полями по слоям (люди, потребности, труд, ресурсы, хранение, ремесло, производство, постройки, улучшения построек, знания, исследования, экономика, логистика, торговля, карта, управление, безопасность, транспорт, энергия, а также слои целей/событий/ограничений в контексте эпохи при необходимости)
- `transition` — `{ to, condition, unlocks: [], checklist: [] }` — **`checklist`** обязателен (непустой массив непустых строк; для K — критерии открытого endgame, см. «Переходы как чеклисты»)
- `timelineSpanYears` — опционально `{ from, to }` в астрономических годах (до н.э. — отрицательные числа); для визуализации пропорции полосы эпох на HTML-карте. Поле `note` допускается (например, пояснение для условного правого конца у эпохи K).
- **Проектный вес эпохи (не игровые характеристики):** `role`, `designWeight`, `gameplayDensity`, `playerTimeShare`, `contentGuideline` — см. раздел «Вес, роль и плотность эпох» ниже и `ARCHITECTURE_SCHEMA.md`.
- **`transition.checklist`:** критерии осмысленного перехода к следующей эпохе (проектные, не лор) — см. «Переходы как чеклисты».

Если renderer ещё не выводит часть полей, данные всё равно можно добавить; HTML не должен падать.

**Игровой каталог** (`gameCatalog`, `gameResources[]`, `gameResearch[]`, `gameTechnologies[]`, `gameEnterprises[]`): именуемые сущности экономики с полями `id`, `title`, `eraFrom`, `eraTo` (буквы эпох A–K), `summary`, `status`. Дополнительно: у ресурсов — `category`; у исследований — `unlocks[]` (теги последствий, не обязательно id других записей); у технологий — `requires[]` (ссылки на id исследований, ресурсов или других технологий — проектное соглашение); у предприятий — `type` (роль узла).

## Вес, роль и плотность эпох

Эпохи **не обязаны** быть одинаково богатыми по числу целей, событий и строк в каталоге. **Нормализация** означает одинаковую **структуру** полей в `data.js`, а не одинаковый **объём** контента.

Поля (проектные метки для планирования и HTML-карты):

- `role` — роль эпохи на пути A→K (обучение, мост, ядро расширения, государственный слой, переход, индустрия, современность, планета, космос, endgame).
- `designWeight` — насколько глубоко **задумывать** эпоху в дизайне: `light` … `core`.
- `gameplayDensity` — ожидаемая плотность решений игрока.
- `playerTimeShare` — ожидаемая доля **реального** игрового времени в эпохе.
- `contentGuideline` — короткий текст: как наполнять и чего избегать (секция «Принцип наполнения» в просмотрщике).

Ориентиры:

- **A** может быть короче по игровому времени: это **вход** и базовые принципы, не бесконечный каменный туториал.
- **B** логично **богаче**: мост от одного персонажа к общей стоянке и ранней общине.
- **C**, **G**, **I**, **J** (и при необходимости **K**) — типичные **core**-слои: сюда уходит основная глубина экономической стратегии или endgame.
- **Каталог игры** (`gameResources`, …) нельзя раздувать быстрее, чем стабилизированы эпохи и системы: каждая новая строка должна иметь `eraFrom` / `eraTo` и ясную связь с системами в смысле архитектуры (см. также правила в `README.md`).

После правок данных запускайте **`node tools/validate-architecture-map.mjs`** из корня репозитория: скрипт проверяет уникальность и полноту **A–K**, цепочку **`transition.to`**, непустые **`transition.checklist`** и `condition`, каталоги и зависимости.

## Переходы как чеклисты

В объекте `transition` поле **`checklist`** — массив коротких строк: что минимально должно быть **спроектировано или реализовано**, чтобы переход к эпохе `to` был осмысленным. Это не лор и не список квестов; 3–7 пунктов обычно достаточно.

- Валидатор требует **непустой** `transition.checklist` для каждой эпохи; пустой массив в data приведёт к ошибке проверки.
- Каждый пункт — проверяемое утверждение уровня дизайна/архитектуры (например: «есть узел X», «игрок понимает Y»).
- Для финальной эпохи K чеклист может описывать критерии **открытой зоны проектирования** (мета-цели, читаемость сверхдолгой игры), а не переход «в следующую букву».

## 9. Статусы

| Код | Смысл |
|-----|--------|
| `prototype` | Уже есть или частично есть в прототипе |
| `designing` | Проектируется сейчас |
| `planned` | Запланировано |
| `later` | Поздняя игра / позже |
| `needs_rework` | Требует пересмотра |
| `blocked` | Нельзя развивать, пока не выполнены зависимости |

Новые статусы добавлять в data и в просмотрщик (лейблы/цвета), не ломая старые.

## 10. Эпохи A и B (зафиксированный смысл)

### A — Позднее выживание

Один персонаж, ручной труд, энергия, груз, огонь, первые материалы, простейшие инструменты. Трансформация: **один персонаж → очаг → малый запас → знакомый радиус → готовность к стоянке.**

В A не вводить рано: население, торговлю, профессии, государство, полноценное земледелие, сложную логистику, сложные здания.

### B — Стоянка и ранняя община

Устойчивая стоянка, общий запас, распорядок, первые люди. Трансформация: **личное выживание → стоянка → малая группа → общий труд → ранняя община.**

Периоды B0–B5 (даты в data.js): устойчивая стоянка → сезонные запасы → люди у огня → распорядок стоянки → ремесленное место → ранняя община.

Переход B → C по смыслу: стоянка → поселение; ручной труд → узлы хозяйства; люди у огня → население; малый запас → складская система; распорядок → приоритеты труда поселения.

## 11. Как работать дальше

1. Найти объект эпохи в `spacegame-architecture-data.js`.
2. Проверить существующие поля.
3. Не ломать структуру; добавлять данные аккуратно.
4. Открыть HTML, проверить консоль, поиск, фильтры.
5. Один коммит — одна понятная задача (например: `docs: expand era C in architecture map`).

Плохо: `docs: update everything`.

## 12. Визуальные улучшения карты (разрешено)

Sticky toolbar, чипы A–K, hash `#era-B`, фильтры A–C / D–F / G–K, компактный/подробный режим, `<details>` для секций, подсветка поиска, оглавление разделов, адаптив. **Сквозные системы:** секция «4. Сквозные системы» — карточки из `systemProgression`, таблица по эпохам с уважением фильтра A–C / D–F / G–K. **Полоса эпох:** ширина карточек пропорциональна `timelineSpanYears` в data (астрономические годы; у K правый конец может быть условным для визуализации). **Полнота структуры карточки:** индикатор чеклиста обязательных полей в data, включая непустой **`transition.checklist`**, и заполненные слои `systems` (не метрика «чем больше целей и событий — тем лучше»). **Матрица:** клик по строке системы подсвечивает эпохи, где система уже «в игре» по `appearsIn`/`becomesCoreIn`. **Каталог игры:** секция с ресурсами, исследованиями, технологиями и предприятиями из отдельных массивов data; фильтр по эпохам пересекает `eraFrom`–`eraTo`. **Vanilla JS**, без роутера, сборщика, canvas-графики, внешних библиотек.

## 13. Как читать карту

Для каждой эпохи — секции: Периоды, Цели, События, Открытия, Ограничения, Системы, Переход. Пустое — не показывать. Много данных — collapsible.

## 14. Критерий качества блока

Блок должен отвечать:

1. На какую эпоху / период опирается?
2. Какую систему затрагивает?
3. Что ограничивает или открывает?
4. Какой игровой смысл?
5. К какому следующему масштабу ведёт?
6. Нужно ли сейчас или позже?

Иначе — не полноценный архитектурный блок; можно TODO или заметка.

## 15. Ближайшая стратегическая последовательность

1. Довести эпоху B в data и в карте.
2. Проработать эпоху C (неолитическое поселение).
3. Привести D–K к единому формату полей.
4. Протянуть систему «Люди» через все эпохи.
5. То же для «Постройки».
6. Исследования / знания.
7. Торговля / логистика.
8. Затем — задачи на игровой код.

## 16. Предупреждение

Не бесконечно углублять каменный век. Цель карты — **весь путь A → … → K** и связь раннего прототипа с будущей экономической стратегией.

## 17. Новые поля и renderer

Если в data появились поля, которые HTML ещё не показывает, renderer обновлять аккуратно: не ломать старые поля; проверять `Array.isArray`; пустые секции скрывать; использовать `<details>`; не дублировать данные вручную в HTML. Пример расширения: `roadmap[].checklist` — маркированный список под карточкой этапа дорожной карты.

## 18. Чеклист перед завершением задачи

1. `SPACEGAME_ARCHITECTURE_MAP.html` открывается локально.
2. Нет ошибок в консоли.
3. Все эпохи отображаются (при фильтре «все»).
4. Поиск и фильтры работают (статус, группа A–C / D–F / G–K, при наличии — вес эпохи `designWeight`).
5. Чипы эпох / hash — по возможности.
6. Компактный/подробный режим — по возможности.
7. `node tools/validate-architecture-map.mjs` завершается без ошибок (после правок `data.js`).
8. Игровой код не изменён.
9. В коммите нет случайных файлов вне области задачи (например, только `docs/architecture/` и при необходимости `tools/` для валидатора).
