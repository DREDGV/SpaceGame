# SpaceGame — правила работы с архитектурной картой игры

Главная задача проекта на этом этапе — не наращивать новые игровые механики в прототипе, а выстроить **архитектурную карту игры**: визуальную, расширяемую, data-driven карту, по которой дальше проектируется вся игра.

## 1. Что мы делаем

Создаётся **рабочий архитектурный инструмент**, а не только документ и не только статичная HTML-страница. Он задаёт проектирование сверху вниз:

**время → эпохи → периоды → системы → зависимости → цели → события → ограничения → переходы → реализация.**

Идея игры: SpaceGame — экономическая стратегия развития цивилизации от позднего выживания (порядка **~10 000 до н.э.**) до планетарной цивилизации, космической экономики и дальнего космоса.

Путь игрока:

**один персонаж → стоянка → ранняя община → поселение → города → государства → индустрия → современная технологическая цивилизация → планетарная экономика → космос → дальние колонии.**

## 2. Главный принцип проектирования

Не проектируем игру от отдельных механик («кнопка», «здание», «ресурс»). Порядок мыслей:

1. Какая эпоха?
2. Какой период внутри эпохи?
3. Какой образ жизни?
4. Какие люди?
5. Какой труд?
6. Какие ресурсы?
7. Какие постройки?
8. Какие знания?
9. Какие ограничения?
10. Какие цели?
11. Какие события?
12. Какой переход к следующей эпохе?

Только после этого — реализация в коде.

## 3. Главный закон времени

Старт исторической шкалы — **примерно ~10 000 до н.э.** Не уходить в миллионы лет и глубокую предысторию.

**Ранние эпохи проходят быстрее, поздние — медленнее, плотнее и дороже по усилиям** (аналогия: добыча Bitcoin — сначала частый прогресс, затем каждый шаг тяжелее).

Ранние эпохи быстро дают ощущение движения: ручной труд → огонь → стоянка → распорядок → малая группа → поселение. Поздняя игра может быть почти бесконечной: индустрия → современность → планета → космос → дальние колонии.

## 4. Текущая шкала эпох (проектировочная, не строгая историческая симуляция)

| Эпоха | Название | Диапазон |
|-------|----------|----------|
| A | Позднее выживание | ~10 000 – 8 500 до н.э. |
| B | Стоянка и ранняя община | ~8 500 – 6 500 до н.э. |
| C | Неолитическое поселение | ~6 500 – 4 500 до н.э. |
| D | Медь, бронза и ранние города | ~4 500 – 1 200 до н.э. |
| E | Железо, античность и средневековье | ~1 200 до н.э. – 1500 н.э. |
| F | Раннее Новое время | 1500 – 1750 |
| G | Индустриальная эпоха | 1750 – 1945 |
| H | Современная технологическая цивилизация | 1945 – 2035 |
| I | Планетарная цивилизация и ранний космос | 2035 – 2150 |
| J | Космическая экономика | 2150 – 3000+ |
| K | Дальний космос | 3000+ |

## 5. Главные файлы карты

Папка: `docs/architecture/`

| Файл | Назначение |
|------|------------|
| `SPACEGAME_ARCHITECTURE_MAP.html` | HTML-просмотрщик |
| `spacegame-architecture-map.css` | Стили |
| `spacegame-architecture-data.js` | **Источник правды** по эпохам, системам, зависимостям |
| `SPACEGAME_ARCHITECTURE_MAP.md` | Текстовый индекс |
| `README.md` | Как открыть карту и редактировать данные |
| `ARCHITECTURE_MAP_RULES.md` | Этот документ — правила и стратегия |

**Правило:** меняются эпохи, периоды, события, цели, ограничения, системы или зависимости — править **`spacegame-architecture-data.js`**. HTML и CSS — только для отображения и навигации.

## 6. Что нельзя делать без прямой команды

- Не трогать игровой код: `web/prototype/`, `game-state.js`, `app.js`, save/load, `SAVE_VERSION`, механики прототипа (очереди, крафт, сбор, экономическое ядро и т.д.).
- Не превращать карту в React/Vue-приложение.
- Не подключать внешние CDN и тяжёлые зависимости.

Карта живёт отдельно от прототипа, пока архитектура не зафиксирована.

## 7. Как устроена карта (data-driven)

```js
window.SPACEGAME_ARCHITECTURE = {
  meta: {},
  eras: [],
  systems: [],
  dependencies: [],
  roadmap: [],
  uiScreens: [],
  gameCatalog: {},
  gameResources: [],
  gameResearch: [],
  gameTechnologies: [],
  gameEnterprises: []
};
```

HTML — просмотрщик. CSS — внешний вид. Data.js — источник правды.

## 8. Целевая структура эпохи в `data.js`

Каждая эпоха **стремится** к полной структуре (пустые массивы и отсутствующие ключи допустимы; секции в HTML не показываются, если данных нет):

- `id`, `title`, `dates`, `summary`, `timePace`, `status`
- `periods[]` — `{ id, title, dates, summary }`
- `goals[]`, `events[]`, `discoveries[]`, `limits[]`
- `systems` — объект с текстовыми полями по слоям (люди, потребности, труд, ресурсы, хранение, ремесло, производство, постройки, улучшения построек, знания, исследования, экономика, логистика, торговля, карта, управление, безопасность, транспорт, энергия, а также слои целей/событий/ограничений в контексте эпохи при необходимости)
- `transition` — `{ to, condition, unlocks: [] }`
- `timelineSpanYears` — опционально `{ from, to }` в астрономических годах (до н.э. — отрицательные числа); для визуализации пропорции полосы эпох на HTML-карте. Поле `note` допускается (например, пояснение для условного правого конца у эпохи K).

Если renderer ещё не выводит часть полей, данные всё равно можно добавить; HTML не должен падать.

**Игровой каталог** (`gameCatalog`, `gameResources[]`, `gameResearch[]`, `gameTechnologies[]`, `gameEnterprises[]`): именуемые сущности экономики с полями `id`, `title`, `eraFrom`, `eraTo` (буквы эпох A–K), `summary`, `status`. Дополнительно: у ресурсов — `category`; у исследований — `unlocks[]` (теги последствий, не обязательно id других записей); у технологий — `requires[]` (ссылки на id исследований, ресурсов или других технологий — проектное соглашение); у предприятий — `type` (роль узла).

## 9. Статусы

| Код | Смысл |
|-----|--------|
| `prototype` | Уже есть или частично есть в прототипе |
| `designing` | Проектируется сейчас |
| `planned` | Запланировано |
| `later` | Поздняя игра / позже |
| `needs_rework` | Требует пересмотра |
| `blocked` | Нельзя развивать, пока не выполнены зависимости |

Новые статусы добавлять в data и в просмотрщик (лейблы/цвета), не ломая старые.

## 10. Эпохи A и B (зафиксированный смысл)

### A — Позднее выживание

Один персонаж, ручной труд, энергия, груз, огонь, первые материалы, простейшие инструменты. Трансформация: **один персонаж → очаг → малый запас → знакомый радиус → готовность к стоянке.**

В A не вводить рано: население, торговлю, профессии, государство, полноценное земледелие, сложную логистику, сложные здания.

### B — Стоянка и ранняя община

Устойчивая стоянка, общий запас, распорядок, первые люди. Трансформация: **личное выживание → стоянка → малая группа → общий труд → ранняя община.**

Периоды B0–B5 (даты в data.js): устойчивая стоянка → сезонные запасы → люди у огня → распорядок стоянки → ремесленное место → ранняя община.

Переход B → C по смыслу: стоянка → поселение; ручной труд → узлы хозяйства; люди у огня → население; малый запас → складская система; распорядок → приоритеты труда поселения.

## 11. Как работать дальше

1. Найти объект эпохи в `spacegame-architecture-data.js`.
2. Проверить существующие поля.
3. Не ломать структуру; добавлять данные аккуратно.
4. Открыть HTML, проверить консоль, поиск, фильтры.
5. Один коммит — одна понятная задача (например: `docs: expand era C in architecture map`).

Плохо: `docs: update everything`.

## 12. Визуальные улучшения карты (разрешено)

Sticky toolbar, чипы A–K, hash `#era-B`, фильтры A–C / D–F / G–K, компактный/подробный режим, `<details>` для секций, подсветка поиска, оглавление разделов, адаптив. **Полоса эпох:** ширина карточек пропорциональна `timelineSpanYears` в data (астрономические годы; у K правый конец может быть условным для визуализации). **Полнота карточки:** индикатор заполненности списков и слоёв `systems`. **Матрица:** клик по строке системы подсвечивает эпохи, где система уже «в игре» по `appearsIn`/`becomesCoreIn`. **Каталог игры:** секция с ресурсами, исследованиями, технологиями и предприятиями из отдельных массивов data; фильтр по эпохам пересекает `eraFrom`–`eraTo`. **Vanilla JS**, без роутера, сборщика, canvas-графики, внешних библиотек.

## 13. Как читать карту

Для каждой эпохи — секции: Периоды, Цели, События, Открытия, Ограничения, Системы, Переход. Пустое — не показывать. Много данных — collapsible.

## 14. Критерий качества блока

Блок должен отвечать:

1. На какую эпоху / период опирается?
2. Какую систему затрагивает?
3. Что ограничивает или открывает?
4. Какой игровой смысл?
5. К какому следующему масштабу ведёт?
6. Нужно ли сейчас или позже?

Иначе — не полноценный архитектурный блок; можно TODO или заметка.

## 15. Ближайшая стратегическая последовательность

1. Довести эпоху B в data и в карте.
2. Проработать эпоху C (неолитическое поселение).
3. Привести D–K к единому формату полей.
4. Протянуть систему «Люди» через все эпохи.
5. То же для «Постройки».
6. Исследования / знания.
7. Торговля / логистика.
8. Затем — задачи на игровой код.

## 16. Предупреждение

Не бесконечно углублять каменный век. Цель карты — **весь путь A → … → K** и связь раннего прототипа с будущей экономической стратегией.

## 17. Новые поля и renderer

Если в data появились поля, которые HTML ещё не показывает, renderer обновлять аккуратно: не ломать старые поля; проверять `Array.isArray`; пустые секции скрывать; использовать `<details>`; не дублировать данные вручную в HTML. Пример расширения: `roadmap[].checklist` — маркированный список под карточкой этапа дорожной карты.

## 18. Чеклист перед завершением задачи

1. `SPACEGAME_ARCHITECTURE_MAP.html` открывается локально.
2. Нет ошибок в консоли.
3. Все эпохи отображаются (при фильтре «все»).
4. Поиск и фильтры работают.
5. Чипы эпох / hash — по возможности.
6. Компактный/подробный режим — по возможности.
7. Игровой код не изменён.
8. В коммите нет случайных файлов вне `docs/architecture/` (если задача только про карту).

## 19. Формат отчёта по задаче

1. Какие файлы изменены.
2. Что изменено по смыслу.
3. Какие новые поля в data.
4. Что улучшено в навигации/чтении.
5. Что проверено.
6. Что не трогалось.
7. Предложенный commit message.
