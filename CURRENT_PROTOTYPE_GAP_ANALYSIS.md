# SpaceGame — анализ разрыва: прототип и дизайн-костяк

Документ сопоставляет **текущий браузерный прототип** (см. `docs/CURRENT_STATE.md`, `CHANGELOG.md`) с рамкой из `SPACEGAME_DESIGN_SPINE.md` и картой эпох **A–G**. Цель — не критиковать отдельные фичи, а показать, **где костяк уже держится**, а где проект «буксует» из-за нехватки сквозной структуры.

---

## Что уже полезно для будущего

- **Экономический каркас ранней стадии:** ресурсы, рецепты, здания, апгрейды, исследования, склады и категории — это ядро, которое масштабируется на более поздние эпохи как «те же сущности, больше связей».
- **Очередь ручного сбора / work lane (Stage 2B):** паттерн «решение → задержка → исполнение» пригоден для труда, производства и транспорта в более поздних эпохах (не только gather).
- **Экономические стадии Stage 1A** (`lone_survivor` → `campfire_camp` → `early_workshop`): хороший прототип **навигатора** «что сейчас главное»; концепцию можно поднять на уровень эпох без привязки только к трём стадиям.
- **Локальная карта лагеря + логистика:** пространственная модель затрат и ноши — задел под карту поселения, регионы и орбиту (разные масштабы, похожая логика).
- **Дневной цикл и ночные нужды:** ранний «ритм общины»; позже тот же слой может стать макро-давлением (город, государство).
- **Знания / озарения / книга знаний:** правильный слой для «закрепления практик» между эпохами.
- **Разнесение UI по модулям:** облегчает появление новых масштабов (панели эпохи, экономическое ядро) без переписывания монолита.

---

## Что временное

- Детальная **микро-логистика одного персонажа** на гексе как *долгосрочный* центр внимания — уместна для A–B, но не должна оставаться главным UI всей primitive-эпохи без моста к «община делает сама».
- Любые **специфические числа** баланса ранней игры без привязки к целевым эпохам — временны до фиксации кривой «A → G».
- **Дублирование данных** между `game-data.js` и split-файлами — технический долг, не часть дизайн-костяка, но мешает быстро наращивать эпохи.

---

## Что переусложнено относительно костяка

- Одновременное развитие **слишком многих ранних UX-поверхностей** (карта, персонаж, кемп, исследования, эконом-ядро, знания) без жёсткого ответа «какой слой ведёт к эпохе C».
- **Primitive-эпоха** в данных уже намечает `early_production` и печь/обжиг, но проектировочный мост «что игрок *перестаёт* делать руками к этому моменту» пока слабее, чем список новых фич.
- Риск **двойного истины** прогресса: вехи эпохи (`eras` / `milestones`), цели `goals`, экономические стадии — без одного «шипа» на документ уровня костяка игрок путается, команда — тоже.

---

## Что надо сохранить

- **Экономическую ось** как главный фильтр фич.
- **Сохранение и версия сейва** как инженерную дисциплину при росте состояния.
- **Карту как пространство затрат**, а не как отдельную мини-игру без производства.
- **Исследования как открытие цепочек**, а не как «+скорость ради скорости».
- **Навигацию «сейчас главное»** (экономическое ядро / этапы) — масштабировать на эпохи, не выкидывать.

---

## Что надо переделать (концептуально, не в коде в этом документе)

- Явно развести **три слоя прогресса**: (1) сюжетный/онбординг момент, (2) экономическая стадия лагеря, (3) эпоха цивилизации — и назначить **один** из них «ведущим на экране» в каждый момент ранней игры.
- Задать **критерии выхода** из primitive не только списком milestone, но и формулировкой «какие рутины игрок больше не обязан повторять».
- Спроектировать **наследование** механик: например, work lane из ранней игры → очередь производства/транспорта позже (один паттерн, разные сущности).

---

## Какие системы прототипа должны стать универсальными

| Система прототипа | Универсальный принцип |
|-------------------|------------------------|
| Ресурсы + склады | Любая эпоха: ограничители и категории |
| Рецепты / цепочки | Производство как граф зависимостей |
| Постройки + автоматизация | Узлы циклов с входом/выходом |
| Исследования | Разблокировка ветвей графа |
| Карта лагеря | Локальный экземпляр «карта = расстояние + риск + запас» |
| Work lane / очередь | «Решение → время → результат» для любого труда |
| Экономические стадии | Шаблон для «фокус эпохи» на любом масштабе |

---

## Что слишком привязано к древней эпохе (риск)

- **Энергия как лимит кликов** — хороша для A, для D+ должна эволюционировать в другие лимиты (бюджет, политика, пропускная способность), иначе жанр застрянет.
- **Персонаж как единственный агент сбора** — уже смягчено очередью; дальше нужен явный мост к «население выполняет базовый сбор».
- **Лексика «пролог / стоянка»** в UI — полезна эмоционально, но не должна доминировать в документации эпохи как будто вся игра «про ледниковый период».

---

## Где риск «кликер с окнами»

- Много **независимых панелей** без одного экономического «якоря» на экране → игрок метается, ощущение кликера.
- Любая новая **награда без новой цепочки или института** → дроп ради дропа.
- **Бесконечная детализация** карты/персонажа без роста масштаба управления → вторая игра внутри игры.

---

## Вывод

Прототип **сильнее всего** на стыке «ранняя экономика + карта + знания». Чтобы не буксовать, недостаёт не «ещё древнего контента», а **жёстко описанного моста** к эпохам C+ и общего **приоритета фич** по матрице систем. Следующий шаг разработки — не наращивать primitive, а **закрепить костяк документами** (сделано) и **согласовать backlog** с `CORE_SYSTEMS_MATRIX.md` и `GAME_LOOP_EVOLUTION.md`.
