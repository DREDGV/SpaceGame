# MAP_RESOURCE_LAYERS_V1_RU

## SpaceGame — ресурсные слои локальной карты V1

Документ для ИИ-агентов и разработчиков.  
Цель — зафиксировать систему ресурсов на локальной карте так, чтобы карта не превращалась в набор ресурсных иконок, а ресурсы существовали как **слои и потенциалы внутри ландшафтных тайлов**.

Документ дополняет:

- `MAP_SYSTEM_V1_RU.md`
- `MAP_DATA_MODEL_V1_RU.md`
- `MAP_UI_V1_RU.md`
- `MAP_ROADS_AND_PATHS_V1_RU.md`
- `MAP_TILE_SEMANTICS_V1_RU.md`

---

## 1. Главная идея

В SpaceGame ресурс не должен быть равен тайлу.

Правильно:

```text
Тайл = место / ландшафтный участок
Ресурсы = слои внутри этого места
```

Неправильно:

```text
Тайл дерева
Тайл камня
Тайл глины
Тайл ягод
```

Правильно:

```text
Лесная опушка:
- ветки
- древесина
- волокно
- ягоды сезонно
- следы животных

Сырой берег:
- вода
- глина
- камыш
- рыба позже

Каменистый склон:
- камни
- острые обломки
- камень для добычи
- руда позже
```

---

## 2. Зачем нужны ресурсные слои

Ресурсные слои нужны, чтобы карта выдержала развитие игры от раннего сбора до промышленной добычи.

Система должна поддерживать:

- ручной сбор веток, камней, волокна;
- сезонные ресурсы;
- истощаемые залежи;
- возобновляемые ресурсы;
- практически неисчерпаемые потоки;
- несколько ресурсов на одном тайле;
- скрытые ресурсы, раскрываемые через знания;
- ресурсные области из нескольких гексов;
- будущую промышленную добычу;
- загрязнение и экологическое влияние позже.

---

## 3. Основное правило для агента

Агент не должен делать карту набором ресурсных иконок.

Ресурсы должны быть связаны с:

```text
terrainType
terrainTags
potentials
resourceLayers
revealedPotentials
availableActions
visibleMarkers
eventSlots
```

Карта сначала показывает ландшафт.  
Ресурс становится видимым только если он раскрыт, найден, очевиден или доступен как действие.

---

## 4. ResourceLayer

`ResourceLayer` — ресурсный слой внутри одного гекса.

Один гекс может иметь несколько `ResourceLayer`.

Пример:

```text
HexTile: Лесная опушка
resourceLayers:
  - branches
  - wood
  - fiber
  - berries
```

Пример:

```text
HexTile: Каменистый берег
resourceLayers:
  - water
  - stone
  - clay
```

---

## 5. Рекомендуемая структура ResourceLayer

```text
ResourceLayer:
  id
  hexId
  resourceType

  depletionType
  amount
  maxAmount
  density
  quality

  regenerationRate
  seasonality

  revealed
  revealRequirement
  extractionRequirement

  extractionDifficulty
  industrialViability

  status[]
```

---

## 6. Пояснение полей ResourceLayer

| Поле | Назначение |
|---|---|
| `id` | уникальный идентификатор слоя |
| `hexId` | гекс, к которому относится слой |
| `resourceType` | тип ресурса |
| `depletionType` | тип истощения/восстановления |
| `amount` | текущий запас |
| `maxAmount` | максимальный запас |
| `density` | насыщенность ресурса на участке |
| `quality` | качество ресурса, опциональное поле |
| `regenerationRate` | скорость восстановления |
| `seasonality` | сезонные правила |
| `revealed` | виден ли слой игроку |
| `revealRequirement` | что нужно для раскрытия |
| `extractionRequirement` | что нужно для добычи |
| `extractionDifficulty` | сложность добычи |
| `industrialViability` | пригодность для промышленной добычи позже |
| `status[]` | статусы слоя |

---

## 7. depletionType

`depletionType` определяет, как ресурс ведёт себя при использовании.

Рекомендуемые значения:

```text
finite
renewable
flow
practically_infinite
```

---

## 8. finite — истощаемые ресурсы

Истощаемые ресурсы уменьшаются при добыче и сами не восстанавливаются в рамках обычной игры.

Примеры:

```text
stone
clay
ore
coal
sand_later
rare_mineral_later
```

Пример:

```text
resourceType: clay
depletionType: finite
amount: 120
maxAmount: 120
```

Использование:

```text
добыли 10 глины → amount уменьшается до 110
```

---

## 9. renewable — возобновляемые ресурсы

Возобновляемые ресурсы могут восстанавливаться со временем, сезонно или при хороших условиях.

Примеры:

```text
branches
wood
fiber
berries
mushrooms
grass
animals_later
fish_later
```

Пример:

```text
resourceType: branches
depletionType: renewable
amount: 12
maxAmount: 20
regenerationRate: slow
```

---

## 10. flow — потоковые ресурсы

Потоковые ресурсы не имеют обычного запаса, но имеют доступность, мощность или пропускную способность.

Примеры:

```text
water_flow
wind
sunlight
river_current_later
```

Пример:

```text
resourceType: water_flow
depletionType: flow
amount: null
density: medium
quality: normal
```

Вода не обязана “заканчиваться”, но может иметь:

- доступность;
- качество;
- загрязнение;
- сезонность;
- риск замерзания;
- риск паводка.

---

## 11. practically_infinite — практически неисчерпаемые ресурсы

Ресурсы, которые в рамках обычного игрового масштаба не должны заканчиваться.

Примеры:

```text
basic_soil
common_water_access
sunlight
ambient_air_later
```

Важно: “практически неисчерпаемый” не значит “всегда полезный”.  
У такого ресурса всё равно могут быть качество, доступность, загрязнение или технологические ограничения.

---

## 12. Несколько ресурсов на одном тайле

Это обязательное правило.

Один тайл может иметь несколько ресурсных слоёв.

Пример:

```text
Лесная опушка:
  branches: renewable
  wood: renewable
  fiber: renewable
  berries: renewable + seasonal
  animal_tracks: event/potential
```

Пример:

```text
Каменистый берег:
  water_flow: flow
  stone: finite
  clay: finite
  reeds: renewable
```

Пример:

```text
Горный ручей:
  water_flow: flow
  stone: finite
  ore_later: finite + hidden
  water_power_later: flow + technology_required
```

---

## 13. density — плотность ресурса

`density` показывает, насколько ресурс насыщен на участке.

Рекомендуемые значения:

```text
very_low
low
medium
high
very_high
```

Плотность влияет на:

- скорость ручного сбора;
- производительность добычи;
- привлекательность промышленного освоения;
- смысл ресурсной области.

Пример:

```text
clay:
  density: high
  amount: 150
```

```text
berries:
  density: low
  amount: 8
```

---

## 14. amount и density — разные вещи

`amount` — сколько всего ресурса доступно.  
`density` — насколько он концентрирован.

Пример:

```text
Много камня, но низкая плотность:
amount: 1000
density: low
```

Это значит: камня много, но добывать его неудобно.

Пример:

```text
Небольшая, но плотная жила:
amount: 150
density: very_high
```

Это значит: общий запас меньше, но добыча эффективная.

---

## 15. quality — качество ресурса

`quality` — спорная и более поздняя характеристика.

В V1 качество нужно заложить в данные, но не делать обязательной активной механикой.

Рекомендуемое правило:

```text
quality хранится, но в V1 почти не влияет на игру.
```

Зачем оставить поле:

- позже качество может влиять на производство;
- автоматизация может требовать лучшего сырья;
- промышленность может сортировать сырьё;
- технологии могут раскрывать разницу качества;
- торговля может учитывать качество;
- сложные рецепты могут требовать ресурс определённого качества.

---

## 16. Почему качество не нужно активно включать в V1

На раннем этапе игроку не нужна лишняя сложность:

```text
камень плохой
камень средний
камень хороший
глина влажная
глина жирная
глина огнеупорная
```

Это может перегрузить пролог и ранний геймплей.

В V1 достаточно:

```text
есть камень
есть глина
есть ветки
есть вода
```

А качество оставить скрытым или условным.

---

## 17. Рекомендуемые значения quality

```text
unknown
poor
normal
good
excellent
special
```

Для V1 почти везде можно ставить:

```text
quality: normal
```

И не показывать игроку, если нет причины.

---

## 18. Когда качество можно раскрывать игроку

Качество ресурса можно раскрывать позже через:

- опыт;
- озарение;
- исследование;
- специалиста;
- инструмент;
- лабораторию;
- промышленный анализ;
- торговлю.

Пример:

```text
Сначала: Камень
Позже: Камень хорошего качества
Ещё позже: Подходит для прочного строительства
```

Пример:

```text
Сначала: Глина
Позже: Пластичная глина
Ещё позже: Подходит для кирпича высокого качества
```

---

## 19. revealRequirement

`revealRequirement` определяет, что нужно, чтобы ресурс стал видимым.

Примеры:

```text
visible_from_start
inspect
insight_sharp_edge
insight_fiber_binding
insight_clay
technology_mining
technology_agriculture
specialist_geologist_later
```

Пример:

```text
resourceType: clay
revealed: false
revealRequirement: insight_clay
```

---

## 20. extractionRequirement

`extractionRequirement` определяет, что нужно для добычи.

Примеры:

```text
none
hands
simple_tool
digging_stick
basket
stone_tool
workshop
mine
industrial_equipment_later
```

Пример:

```text
branches:
  extractionRequirement: hands
```

```text
clay:
  extractionRequirement: digging_stick
```

```text
ore:
  extractionRequirement: mining_technology_later
```

---

## 21. extractionDifficulty

`extractionDifficulty` — сложность добычи.

Рекомендуемые значения:

```text
very_easy
easy
medium
hard
very_hard
```

Влияет на:

- время сбора;
- расход энергии;
- потребность в инструментах;
- производительность работников;
- смысл улучшения дороги/логистики.

---

## 22. industrialViability

`industrialViability` показывает, есть ли смысл промышленно осваивать ресурсную область позже.

Рекомендуемые значения:

```text
none
low
medium
high
strategic
```

Пример:

```text
Небольшие камни у лагеря:
industrialViability: none
```

```text
Крупный каменистый склон:
industrialViability: high
```

```text
Рудная жила:
industrialViability: strategic
```

В V1 это поле можно не показывать игроку.

---

## 23. ResourceArea

`ResourceArea` — ресурсная область из нескольких гексов.

Она нужна для будущего промышленного сбора, добычи, полей, карьеров, лесозаготовок и рудников.

```text
ResourceArea = группа гексов, связанных общим ресурсным смыслом
```

Примеры:

```text
Глиняный берег
Каменистый склон
Лесной массив
Плодородная низина
Рудная жила
Рыбный участок реки
```

---

## 24. Структура ResourceArea

```text
ResourceArea:
  id
  name
  resourceType
  hexIds[]

  depletionType
  totalAmount
  averageDensity
  averageQuality

  discovered
  surveyLevel
  industrialViability

  status[]
```

---

## 25. Зачем нужна ResourceArea

Без ResourceArea промышленная добыча будет странной: один карьер на одном гексе, одно поле на одном гексе, один лесоруб на одном дереве.

Правильнее:

```text
поле = область из нескольких плодородных гексов
карьер = область из нескольких каменистых гексов
лесозаготовка = область лесных гексов
рудник = область рудных гексов
```

В V1 можно не реализовывать полноценные ResourceArea, но данные должны быть совместимы с их добавлением.

---

## 26. Ресурсные области в UI

В обычном режиме ресурсные области не должны перегружать карту.

Рекомендуемое отображение:

### Обычный режим

- виден ландшафт;
- видны только раскрытые маленькие маркеры;
- нет больших ресурсных иконок.

### Режим ресурсов

- подсвечиваются ресурсные области;
- показывается контур области;
- показывается плотность;
- показывается раскрытая информация;
- скрытые ресурсы не раскрываются без знаний.

Пример:

```text
Глиняный берег
Плотность: высокая
Запас: средний
Качество: неизвестно
```

## 27. Связь с MAP_TILE_SEMANTICS

`MAP_TILE_SEMANTICS_V1_RU.md` описывает смысл тайла.

Этот документ описывает ресурсную механику внутри этого смысла.

Связь:

```text
terrainTags → potentials → resourceLayers → availableActions
```

Пример:

```text
terrainTags: [wet, near_water, soft_soil]
potentials: [water, clay, reeds]
resourceLayers:
  - water_flow
  - clay
  - reeds
availableActions:
  - gather_water
  - dig_clay после раскрытия
```

---

## 28. Ресурс не обязан быть видимым

У `ResourceLayer` есть поле:

```text
revealed
```

Если:

```text
revealed = false
```

то ресурс:

- не показывается крупной иконкой;
- не доступен как действие;
- может быть упомянут только как скрытый потенциал в данных;
- может быть раскрыт через осмотр, озарение или технологию.

## 31. Как сбор ресурса должен работать в V1

При действии сбора:

```text
1. Найти соответствующий ResourceLayer.
2. Проверить, что он revealed = true.
3. Проверить extractionRequirement.
4. Рассчитать количество сбора.
5. Уменьшить amount, если ресурс finite или renewable.
6. Добавить ресурс игроку.
7. При необходимости обновить visibleMarkers/status.
8. Перемещение к ресурсу должно увеличивать footTraffic на RoadLink.
```

---

## 32. Формула сбора V1

Для V1 достаточно простой формулы:

```text
collectedAmount = baseAmount × densityModifier × toolModifier
```

Где:

```text
densityModifier:
very_low = 0.5
low = 0.75
medium = 1.0
high = 1.25
very_high = 1.5
```

`quality` в V1 не использовать в формуле, если нет отдельного задания.

---

## 33. Восстановление renewable ресурсов

Для V1 можно сделать просто:

```text
каждый игровой день / цикл:
  amount += regenerationRate
  amount не больше maxAmount
```

Если в проекте ещё нет времени/дней, восстановление можно отложить.

Но depletionType должен быть заложен.

---

## 34. Практически неисчерпаемые ресурсы и flow

Для `flow` и `practically_infinite` ресурсы не обязательно уменьшать при использовании.

Пример:

```text
gather_water не обязан уменьшать water_flow.amount
```

Но позже могут появиться ограничения:

- загрязнение;
- сезон;
- замерзание;
- низкий уровень воды;
- засуха;
- пропускная способность.

---

## 35. Отображение ресурсов на карте

Агент должен соблюдать правило:

```text
В обычном режиме карта показывает ландшафт, а не склад ресурсных иконок.
```

Разрешено:

- маленький маркер раскрытого ресурса;
- подсказка при наведении;
- список в панели тайла;
- отдельный режим ресурсов;
- временный маркер события.

Запрещено:

- большие постоянные иконки всех ресурсов;
- раскрытие скрытых ресурсов до знания;
- замена ландшафтного арта ресурсной иконкой.

---

## 36. Панель тайла для ресурсов

Панель тайла должна показывать только раскрытые ресурсы.

Пример:

```text
Ресурсы:
- Вода: доступна
- Глина: неизвестно / не раскрыто
```

Лучше в V1 скрытые ресурсы не показывать вообще, чтобы не спойлерить.

После раскрытия:

```text
Ресурсы:
- Вода: доступна
- Глина: средняя плотность
```

---

## 37. Качество в UI

В V1 `quality` не выводить в UI по умолчанию.

Исключения:

- ресурс специально раскрыт через событие;
- ресурс имеет `quality: special`;
- игрок получил знание/технологию анализа;
- будущий режим ресурсов включён.

По умолчанию:

```text
качество скрыто
```

Это снижает перегруз.

---

## 38. Что агент должен сделать

Агент должен:

1. Ввести `resourceLayers[]` или совместимую структуру.
2. Разрешить несколько ресурсов на одном тайле.
3. Добавить `depletionType`.
4. Добавить `amount` / `maxAmount` там, где нужно.
5. Добавить `density`.
6. Добавить `quality`, но не использовать активно в V1.
7. Добавить `revealed`.
8. Связать доступные действия с раскрытыми ресурсными слоями.
9. Убрать модель “один тайл = один ресурс”.
10. Не показывать все ресурсы крупными иконками на карте.

## 39. Что агент НЕ должен делать

Агенту запрещено:

- делать `resourceType` единственным ресурсом тайла;

- показывать скрытые ресурсы игроку;
- делать качество ресурса обязательной активной механикой V1;
- усложнять пролог сортами ресурсов без необходимости;
- добавлять промышленные ResourceArea как полноценную систему без отдельной задачи;

## 40. Минимальный результат V1

Система ресурсных слоёв V1 считается готовой, если:

- один гекс может иметь несколько ресурсных слоёв;
- ресурсы имеют тип истощения;
- часть ресурсов может быть скрыта;
- доступные действия зависят от раскрытых ресурсов;
- ресурсы не являются главной визуальной сущностью тайла;
- есть плотность ресурса;
- качество ресурса заложено, но не перегружает UI;
- данные совместимы с будущими ResourceArea.

## 41. Главная формула документа

```text
Тайл — это место.
Место имеет ресурсные слои.
Ресурсный слой имеет запас, плотность, тип восстановления и раскрытие.
Качество ресурса заложено, но в V1 почти не используется.
Несколько тайлов позже могут образовать ресурсную область.
```

Главный запрет:

```text
Не делать карту набором ресурсных иконок и не считать, что у тайла может быть только один ресурс.
```
