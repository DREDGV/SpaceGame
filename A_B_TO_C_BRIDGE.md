# SpaceGame — мост от прототипа A–B к эпохе C

Документ описывает **проектировочный** переход от текущего слоя прототипа (**эпохи A–B**) к **эпохе C — Поселение и ремесленный город**. Не добавляет игровых механик в код и не детализирует отдельные системы (торговля, земледелие, охота, симуляция жителей как отдельный симулятор) — только рамку решений и критерии.

**Канон эпох, дат и переходов** (включая формулировку B→C и заголовок эпохи C в данных: «Неолитическое поселение») — `docs/architecture/spacegame-architecture-data.js`; визуализация — `docs/architecture/SPACEGAME_ARCHITECTURE_MAP.html`. В этом документе название C оставлено в проектировочной формулировке «поселение / ремесленный город»; смысл совпадает с полем `title`/`summary` эпохи C в data.

Канонические названия эпох A–G в сжатой линейке: см. `ERA_PROGRESSION_MAP.md` и таблицу соответствия A–G ↔ A–K в начале того файла.

---

## 1. Как текущий прототип A–B переходит к C

**A–B сегодня:** один управляемый фокус (персонаж + локальный лагерь + очередь работ + ранние цепочки), экономические стадии Stage 1A и primitive-вехи закрепляют «костёр → мастерская → знания».

**C начинается** не с новой мини-игры, а с момента, когда:

- производство и запасы описываются **сетью зданий и цехов**, а не только «герой + несколько автоматизаций»;
- **узкие места** чаще лежат в **складе, пропускной способности, распределении труда между узлами**, а не в каждом клике сбора;
- карта лагеря остаётся **пространством затрат**, но игрок чаще принимает решения уровня **«какой участок / какой цех / какой приоритет очереди»**, а не «каждый шаг ног персонажа».

Мост — это **сдвиг доминанты интерфейса и данных** от «агент на гексе» к «поселение как хозяйство», без смены жанра (см. `GAME_LOOP_EVOLUTION.md`).

---

## 2. Что игрок перестаёт делать руками

По мере закрепления моста (критерии ниже) игрок **не обязан** постоянно:

- инициировать **каждую партию** базового сбора сырья, если для узла поселения уже заданы приоритеты и доступность труда/жителей (см. п. 3 — без отдельной механики «симулятор каждого»);
- микроуправлять **каждым** выходом на тайл, когда маршрут и правила добычи для поселения стабильны;
- держать в голове **только** энергию персонажа как главный лимитер всех цепочек — лимиты смещаются в сторону **пропускной способности поселения** (склады, цеха, очереди).

Игрок **по-прежнему** может вмешиваться вручную (override), но **норма** эпохи C — решения уровня поселения, а не постоянный микроклик.

---

## 3. Какие ручные работы становятся «работами жителей»

Здесь **«жители»** — проектировочная абстракция: совокупность доступного труда поселения (занятость, приоритеты, простые роли), **без** требования в этом документе ввести отдельную симуляцию каждого NPC.

Примеры переноса смысла (не механик):

| В A–B сейчас (смысл) | В C (смысл) |
|----------------------|-------------|
| Игрок ставит в очередь каждый ручной сбор | Тот же паттерн очереди относится к **заданиям поселения** на добычу/нос по правилам приоритета |
| Персонаж носит груз с участка | **Труд поселения** закрывает стандартные носки/разгрузки по открытым маршрутам; персонаж — для особых выездов |
| Энергия ограничивает число действий «своими руками» | Давление смещается на **занятость, смены, простой** узлов (без обязательной детализации смен в первом макете C) |

Важно: это **не** внедрение охоты, земледелия или торговли как новых систем в данном документе — только перераспределение **того же экономического смысла** (труд, время, запас) на слой поселения.

## 4. Какие системы текущего прототипа наследует эпоха C

| Система прототипа | Наследование в C |
|-------------------|------------------|
| Ресурсы и склады | Больше узлов входа/выхода, **категории** остаются осью |
| Рецепты / цепочки | Те же графы зависимостей; больше звеньев «цех ↔ склад» |
| Постройки и автоматизация | Те же циклы зданий; приоритеты привязаны к **узлам поселения** |
| Исследования / знания | Те же разблокировки; веха C — «ремесленный город» как новые цепочки, не новый тип прогресса |
| Карта лагеря | Тот же принцип «расстояние и риск»; масштаб UI может оставаться локальным, смысл — **логистика поселения** |
| Work lane / очередь | Тот же паттерн «решение → время → результат» для **работ поселения**, не только gather |
| Экономическое ядро / навигация «главное» | Расширяется на **узкие места поселения** (склад, цех, очередь), а не только на руки героя |

## 5. Какие UI-блоки становятся второстепенными

Проектировочно (без правок кода в этом PR):

- **Постоянный центр** экрана смещается с «карта + персонаж как главный способ добыть» к **«поселение / цепочка / очередь»** — карта и персонаж остаются важны, но как **второй слой**: проверка маршрута, исключение, сюжетный выезд.
- Детальные **микро-подсказки** по одному гексу не должны оставаться единственным способом понять «что делать дальше» — экономический якорь должен читаться из **блока поселения / цехов / складов** (концептуально тот же экономический UI, другой акцент).
- **Энергия персонажа** как главный «глобальный таймер всех действий» — второстепенна относительно **пропускной способности и запасов поселения** в типичной сессии C (персонаж не исчезает).

## 6. Критерии перехода к C

Переход **задокументированно** считается возможным, когда выполняются **все** пункты:

1. **Производство как привычка:** игрок стабильно держит хотя бы одну **повторяемую** цепочку (не разовый крафт «ради галочки»), соответствующую вехам выхода из B (см. `ERA_PROGRESSION_MAP.md`, эпоха B).
2. **Знания закрепили практику:** открыты решения, которые объясняют, *почему* поселение может брать базовые задачи на себя (без новой механики — через уже существующие исследования/озарения в данных).
3. **Мост по труду:** в проектировочном смысле определено, **какие классы ручных действий A–B** перестают быть обязательными для прогресса (список из п. 2 согласован с дизайн-ревью).
4. **UI-намерение:** зафиксирован «ведущий слой» для экрана в C (поселение vs персонаж) — см. `CURRENT_PROTOTYPE_GAP_ANALYSIS.md` и этот документ.

До выполнения критериев расширять контент **только** внутри A–B без моста — риск бесконечной primitive-полировки.

## 7. Минимальный макет C перед реализацией

**До кодинга** эпохи C должны существовать (как артефакты дизайна, не как код):

1. **Схема поселения** из 5–9 узлов (здание / склад / «ворота» / цех) с потоками вход–выход (стрелки на бумаге или в диаграмме).
2. **Список из 3–5 цепочек** «сырьё → полуфабрикат → потребление/стройка», которые **продолжают** текущие рецепты primitive, а не переписывают экономику с нуля.
3. **Одна таблица приоритетов** очереди поселения (что важнее: еда, топливо, ремонт, запас для стройки) — логика уровня правил, не UI.
4. **Один экран-макет** (wireframe) «главное в C»: где склад, где очередь, где карта — без реализации.
5. **Явный список «не делаем до C»** (см. п. 8) — подписанный вместе с макетом.

---

## 8. Какие фичи запрещено добавлять до появления этого моста

До утверждения макета C (п. 7) и критериев перехода (п. 6) **не следует** добавлять в ближайший бэклог:

- новые **игровые оси** (отдельная торговля, земледелие, охота как самостоятельные петли), если они не обслужуют **тот же** мост к цепочкам поселения;
- новые **экраны-центры**, конкурирующие с экономическим якорем, без места в матрице эпох A–K в `docs/architecture/`;
- **дальнейшую детализацию** только карты/персонажа, если она не снимает рутину A–B и не ведёт к узлам поселения;
- «ещё одну эпоху» или режим в данных, который **размывает** primitive / early_production без связи с C.

Разрешено: документы, макеты, согласование критериев, рефакторинг **концепции** прогресса (три слоя: онбординг / эконом-стадия / эра) — без новых механик в коде по этому документу.
