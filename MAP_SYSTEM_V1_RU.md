# MAP_SYSTEM_V1_RU

## SpaceGame — система локальной карты V1

Документ для ИИ-агентов и разработчиков. Его цель — зафиксировать базовую архитектуру карты SpaceGame перед внедрением в проект.

Этот документ описывает только первую реалистичную версию системы карт. Он не должен превращаться в полную финальную архитектуру всей игры. Главная задача V1 — получить рабочую локальную карту, на которой можно строить, развивать участки, соединять их дорогами и сохранять состояние.

---

## 1. Главная цель V1

В первой версии нужно реализовать **локальную flat-top hex-карту** как основной рабочий игровой слой.

Локальная карта должна стать местом, где игрок:

- видит территорию поселения;
- выбирает участки;
- строит первые объекты;
- развивает лагерь;
- использует ресурсы местности;
- соединяет участки тропами/дорогами;
- видит базовые характеристики гекса;
- открывает панель выбранного участка;
- сохраняет и загружает состояние карты.

В V1 не нужно реализовывать региональную карту, планету, орбиту, космос, сложную политику, крупные промышленные комплексы или свободное строительство.

---

## 2. Принятое ключевое решение: ориентация гексов

В SpaceGame используются **flat-top hex**.

Это означает:

- верхняя и нижняя стороны гекса горизонтальные;
- все визуальные схемы, промты, UI-макеты и код должны учитывать именно эту ориентацию;
- нельзя случайно использовать pointy-top hex, где вершина направлена вверх.

Это обязательное решение для всех будущих задач, связанных с картой.

---

## 3. Масштаб локальной карты

Для локальной карты принимается условный масштаб:

```text
1 локальный гекс ≈ 20 м
```

Это не инженерная точность, а игровая условность.

Один гекс следует воспринимать как **участок территории**, а не как точное место под одно физическое здание.

---

## 4. Главная модель гекса

В V1 гекс — это не просто клетка и не просто фон.

Гекс — это **участок территории**.

У участка есть:

- тип местности;
- вместимость;
- занятое место;
- тип участка;
- список построек;
- ресурсный потенциал;
- базовые качества;
- владелец/контроль;
- дорожные связи;
- статус.

Базовая формула:

```text
Гекс = участок территории
Участок имеет вместимость
Постройки занимают часть вместимости
На карте показывается тип участка
Подробности показываются в панели участка
```

---

## 5. Вместимость гекса

Вместимость гекса показывает, сколько условного строительного пространства доступно внутри участка.

Базовое значение для хорошего ровного участка:

```text
capacity = 20
```

Фактическая вместимость зависит от местности.

Пример стартовых значений:

| Тип местности | Вместимость |
|---|---:|
| Ровная поляна | 20 |
| Плодородная земля | 20 |
| Сухая равнина | 18 |
| Берег / ручей | 16 |
| Лес | 14 |
| Холм | 12 |
| Каменистый участок | 10 |
| Склон | 8 |
| Болото | 5 |
| Вода | 0–4 |

В V1 достаточно реализовать несколько типов местности, например:

- поляна;
- лес;
- берег/ручей;
- каменистый участок;
- плодородная земля;
- глина;
- вода.

---

## 6. Постройки внутри гекса

Постройки занимают часть вместимости участка.

Пример:

```text
Участок: складской
Вместимость: 16
Постройки:
- Склад: 5
- Навес: 2
Занято: 7
Свободно: 9
```

В V1 постройка не должна иметь сложный footprint на несколько гексов. Она занимает условное количество места внутри гекса.

Это сделано специально, чтобы:

- не усложнять размещение;
- не бороться с кривым визуальным footprint;
- не превращать игру в градостроительный редактор;
- сохранить читаемость карты;
- упростить реализацию для браузерного проекта.

---

## 7. Что показывать на самой карте

На локальной карте не нужно показывать весь состав гекса буквально.

На карте гекс должен показывать:

1. тип участка;
2. основной визуальный образ;
3. занятость, если это нужно;
4. важный статус;
5. подключение к дороге, если это важно.

Пример отображения:

```text
Складской участок
7/16
Подключён
```

Полный список построек, улучшений, рабочих и проблем должен отображаться в панели выбранного участка.

---

## 8. Типы участков V1

Для первой версии достаточно следующих типов участков:

| Тип участка | Назначение |
|---|---|
| Пустой | участок без застройки |
| Лагерный | стартовый центр, костёр, базовое управление |
| Жилой | укрытия, жильё, население |
| Складской | хранение ресурсов |
| Ремесленный | мастерская, обработка ресурсов |
| Сельскохозяйственный | поле, еда, выращивание |
| Добывающий | камень, глина, руда позже |
| Оборонительный | пост, сторожевая точка |
| Лесной / ресурсный | древесина, сбор, охота позже |
| Водный / береговой | вода, рыбалка, переправа позже |

Позже можно добавить:

- торговый;
- промышленный;
- энергетический;
- научный;
- портовый;
- космический;
- специальный.

## 9. Минимальные постройки V1

Для первой версии можно использовать такой стартовый набор:

| Постройка | Тип участка | Стоимость места | Назначение |
|---|---|---:|---|
| Костёр | лагерный | 2 | стартовый центр |
| Укрытие | жилой | 4 | жильё |
| Склад | складской | 5 | хранение ресурсов |
| Мастерская | ремесленный | 6 | производство простых предметов |
| Поле | сельскохозяйственный | 8 | еда |

| Каменоломня | добывающий | 6 | добыча камня |
| Сторожевой пост | оборонительный | 3 | контроль/безопасность |

Цифры предварительные. Их можно менять при балансировке.

## 10. Панель выбранного участка

При клике по гексу должна открываться панель участка.

Минимальное содержимое панели:

```text
Название / тип участка
Тип местности
Вместимость: занято / доступно
Постройки
Свободное место
Ресурс участка
Качества участка
Дорожные подключения
Доступные действия
Статусы / проблемы
```

Пример:

```text
Участок: Складской
Местность: Ровная поляна
Вместимость: 7 / 20

Постройки:
- Склад: 5
- Навес: 2

Подключения:
- Восток: тропа
- Юго-восток: нет

Действия:
[Построить]
[Улучшить]
[Проложить тропу]
[Очистить участок]
```

---

## 11. Внутренний экран участка

В V1 не нужно реализовывать полноценный внутренний экран участка с 5×4 слотами.

Решение для V1:

```text
Клик по гексу → панель участка
```

Идею внутренней схемы участка 5×4 можно оставить на будущее.

Важно: не делать мини-гексы внутри гекса в первой версии.

---

## 12. Апгрейды участка

Гексы можно улучшать, но в V1 это должно быть очень ограниченно.

Минимальные апгрейды V1:

| Апгрейд | Эффект |
|---|---|
| Расчистка | увеличивает доступную вместимость лесного/заросшего участка |
| Выравнивание | повышает пригодность к строительству |
| Тропа / дорога | улучшает логистику между участками |

Позже можно добавить:

- фундамент;
- дренаж;
- террасирование;
- ирригацию;
- подведение воды;
- подведение энергии;
- экологическую очистку;
- защиту от паводка;
- укрепление берега.

---

## 13. Дороги и связи между гексами

В логике игры дорога — это связь между двумя соседними гексами.

```text
RoadLink = связь Hex A ↔ Hex B
```

У связи могут быть поля:

```text
fromHexId
toHexId
type
level
capacity
condition
ownerId
traffic
```

Типы связи для V1:

- нет связи;
- тропа;
- дорога.

Позже:

- железная дорога;
- ЛЭП;
- трубопровод;
- канал;
- магистраль.

Визуал дорог в V1 должен быть простым и читаемым. Не нужно сразу решать финальный художественный стиль дорог.

Рекомендуемый временный вариант:

```text
дорога рисуется как простой сегмент между соседними гексами и совпадает с направлением движения юнитов/ресурсов
```

Важно: визуал дороги не должен ломать арт тайла.

## 14. Режимы карты V1

Для первой версии достаточно 2–3 режимов:

### 1. Обзор

Показывает:

- гексы;
- типы участков;
- постройки;
- базовые статусы;
- дороги.

### 2. Строительство

Показывает:

- доступные участки;
- вместимость;
- где можно построить выбранную постройку;
- причины, почему нельзя строить.

### 3. Логистика

Можно сделать позже в рамках V1.1.

Показывает:

- дороги;
- подключения;
- складские связи;
- маршруты.

В V1 не нужно показывать все слои одновременно.

---

## 15. Зоны и комплексы

В V1 не нужно реализовывать полноценные крупные комплексы.

Но архитектурно нужно понимать будущую модель:

```text
Зона = группа похожих гексов
Комплекс = группа связанных разных гексов
```

Примеры зон:

- поле;
- лесозаготовка;
- карьер;
- пастбище.

Примеры комплексов:

- порт;
- завод;
- космодром;
- промышленный район.

В первой версии можно не делать зоны как отдельную сущность. Достаточно, чтобы несколько соседних гексов могли иметь один и тот же тип, например сельскохозяйственный.

---

## 16. Территории и контроль

В V1 достаточно минимального контроля:

- стартовый лагерь контролирует ближайшие участки;
- строить можно только в контролируемой зоне;
- дальние участки могут быть неизвестными или неконтролируемыми.

Минимальные поля:

```text
ownerId
controllerId
controlled: true/false
discovered: true/false
```

Политические владения, спорные территории, войны и фракции — позже.

---

## 17. Производительность и размер карты

Для первой версии рекомендуется стартовать с карты:

```text
32×32 hex
```

Это даст 1024 гекса, что достаточно для прототипа.

Позже можно перейти к:

```text
48×48
64×64
```

Но сначала важнее получить рабочую механику, а не огромную карту.

## 20. Что не делать в V1

Не реализовывать:

- свободное строительство без тайлов;
- сложные footprint зданий на несколько гексов;
- микрогексы внутри гекса;
- полноценную внутреннюю карту участка;
- крупные промышленные комплексы;
- региональную карту;
- карту планеты;
- орбиту;
- космос;
- сложную экологию;
- электросети;
- трубопроводы;
- железные дороги;
- войны и фракционные границы.

Эти системы можно закладывать в архитектуру, но не делать на первом этапе.

---

## 21. Критерии готовности V1

Система локальной карты V1 считается готовой, если:

- отображается flat-top hex-карта;
- можно выбрать гекс;
- открывается панель участка;
- у гекса есть местность, вместимость и тип участка;
- можно построить несколько базовых построек;
- постройки занимают вместимость участка;
- нельзя строить, если не хватает места;
- есть минимум два типа дорожной связи: тропа и дорога;
- дороги соединяют соседние гексы;
- состояние карты сохраняется и загружается;
- UI остаётся читаемым и не перегруженным.

---

## 22. Итоговая формула V1

```text
Локальная карта SpaceGame V1 =
flat-top hex grid
+ гекс как участок территории
+ вместимость участка
+ постройки внутри участка
+ панель выбранного участка
+ простые дороги между соседними гексами
+ базовое сохранение
```

Главный принцип:

```text
Сначала рабочая карта. Потом красота, сложные режимы и большие масштабы.
```
