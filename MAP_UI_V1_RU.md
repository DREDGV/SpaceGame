# MAP_UI_V1_RU

## SpaceGame — UI локальной карты V1

Документ для ИИ-агентов и разработчиков.  
Цель — описать интерфейс первой версии локальной карты SpaceGame.

Документ дополняет:

- `MAP_SYSTEM_V1_RU.md`
- `MAP_DATA_MODEL_V1_RU.md`

Главный принцип UI V1:

```text
Карта должна быть понятной, рабочей и не перегруженной.
Сначала читаемость и управление. Потом красота, анимации и сложные режимы.
```

---

## 1. Назначение UI локальной карты

Локальная карта — основной повседневный экран игрока в ранней версии SpaceGame.

Через неё игрок должен:

- видеть территорию;
- выбирать гексы;
- понимать типы участков;
- видеть базовые постройки;
- открывать панель участка;
- строить первые объекты;
- видеть вместимость участка;
- прокладывать тропы/дороги;
- понимать, какие участки доступны, заняты, свободны или заблокированы.

В V1 не нужно делать сложный интерфейс планетарной карты, региона, орбиты или космоса.

---

## 2. Общая структура экрана

Рекомендуемая структура экрана локальной карты:

```text
┌──────────────────────────────────────────────┐
│ Верхняя панель ресурсов / статуса             │
├───────────────┬──────────────────────┬───────┤
│ Левая панель  │     Локальная карта  │ Правая│
│ режимов       │     flat-top hex     │ панель│
│               │                      │ участка│
├───────────────┴──────────────────────┴───────┤
│ Нижняя панель действий / подсказок             │
└──────────────────────────────────────────────┘
```

Минимально нужны:

- верхняя панель ресурсов;
- центральная область карты;
- правая панель выбранного гекса;
- простая нижняя панель действий.

Левую панель режимов можно сделать сразу или позже.

---

## 3. Центральная карта

Центральная часть экрана показывает локальную flat-top hex-карту.

Обязательные требования:

- гексы должны быть flat-top;
- верхняя и нижняя стороны гекса горизонтальные;
- сетка должна быть читаемой;
- гексы не должны выглядеть как pointy-top;
- игрок должен понимать, какой гекс выбран;
- тип участка должен считываться визуально;
- карта не должна быть перегружена всеми слоями одновременно.

---

## 4. Ориентация гексов

Весь UI должен использовать только flat-top hex.

Правильно:

```text
⬡ с горизонтальной верхней и нижней стороной
```

Неправильно:

```text
⬢ с вершиной сверху
```

Если агент генерирует визуалы, компоненты или координатную сетку, он обязан учитывать flat-top orientation.

---

## 5. Размер локальной карты в UI

Для V1 рекомендуется начинать с карты:

```text
32×32 hex
```

Но на экране одновременно не обязательно показывать всю карту.

UI должен поддерживать:

- панорамирование;
- масштабирование позже;
- центрирование на лагере;
- выделение выбранного гекса.

Если масштабирование ещё не реализовано, стартовая версия может показывать ограниченную область вокруг лагеря.

---

## 6. Что видно на гексе в обычном режиме

В обычном режиме на гексе должно быть видно:

1. тип местности;
2. тип участка;
3. основной визуальный образ участка;
4. выбранность, если гекс выбран;
5. занятость, если гекс уже используется;
6. дорожные связи, если есть.

Не нужно показывать:

- все внутренние постройки;
- все показатели качества;
- все статусы;
- все слои логистики;
- все возможные маршруты;
- политические или экологические слои.

---

## 7. Визуальные типы гексов V1

Для первой версии нужны простые, различимые визуальные состояния.

### 7.1. Типы местности

| terrainType | Как должен выглядеть |
|---|---|
| `plain` | трава / ровная поляна |
| `fertile_plain` | более зелёная плодородная земля |
| `forest` | лесной участок |
| `river_bank` | берег / ручей рядом |
| `stone` | камни / серый грунт |
| `clay` | глина / рыжеватая земля |
| `water` | вода |
| `hill` | холмистый или неровный участок |
| `swamp` | болотистый участок |

### 7.2. Типы участков

| plotType | Визуальный смысл |
|---|---|
| `empty` | свободный участок |
| `camp` | лагерь / костёр / центр |
| `residential` | укрытия / жильё |
| `storage` | склад / ящики / навес |
| `craft` | мастерская / материалы / дымок |
| `agriculture` | поле / грядки |
| `extraction` | добыча / яма / камни |
| `defense` | пост / вышка / укрепление |
| `resource` | природный ресурс |
| `water` | водный участок |

В V1 допустимо использовать простые иконки и цветовые акценты вместо полноценного арта.

---

## 8. Индикатор вместимости

Для занятых участков желательно показывать компактный индикатор:

```text
7/20
12/16
```

Но не обязательно показывать его на каждом гексе постоянно.

Рекомендуемое поведение:

| Состояние | Что показывать |
|---|---|
| обычный обзор | только если участок занят или выбран |
| наведение | показать вместимость во всплывающей подсказке |
| выбранный гекс | показать в правой панели |
| режим строительства | показать вместимость на доступных участках |

---

## 9. Выбор гекса

При клике по гексу:

- гекс выделяется рамкой/подсветкой;
- открывается правая панель участка;
- в панели показывается подробная информация;
- карта не должна терять фокус;
- повторный клик по пустой области может снимать выделение.

Визуал выбранного гекса:

- мягкая светлая рамка;
- лёгкое свечение;
- не слишком яркая заливка;
- не перекрывать арт тайла.

---

## 10. Hover / наведение

При наведении на гекс можно показывать краткую подсказку.

Пример:

```text
Лесной участок
Вместимость: 14
Ресурс: древесина
Строительство: среднее
```

Не нужно показывать большую панель или внутреннюю схему участка при каждом наведении.

Важно: hover не должен мешать игре на мобильных устройствах. Основная информация должна быть доступна по клику.

---

## 11. Правая панель участка

Правая панель — главный инструмент работы с выбранным гексом.

Минимальные блоки панели:

```text
Название участка
Тип местности
Тип участка
Вместимость
Постройки
Ресурсы
Качества
Дороги / подключения
Действия
Статусы
```

---

## 12. Пример панели пустого гекса

```text
Участок: Пустой
Местность: Ровная поляна
Вместимость: 20 / 20

Качества:
Строительство: высокое
Сельское хозяйство: среднее
Добыча: низкая

Действия:
[Построить]
[Назначить тип участка]
[Проложить тропу]
```

---

## 13. Пример панели занятого гекса

```text
Участок: Складской
Местность: Ровная поляна
Вместимость: 7 / 20
Свободно: 13

Постройки:
- Склад: 5
- Навес: 2

Подключения:
- Восток: тропа
- Юго-восток: дорога

Статус:
Подключён к лагерю

Действия:
[Построить]
[Улучшить участок]
[Проложить дорогу]
[Снести постройку]
```

---

## 14. Панель построек внутри участка

Постройки внутри участка в V1 показываются списком.

Пример:

```text
Постройки:
1. Склад
   Место: 5
   Уровень: 1
   Состояние: 100%

2. Навес
   Место: 2
   Уровень: 1
   Состояние: 100%
```

В V1 не нужно показывать внутреннюю сетку 5×4.

---

## 15. Действие “Построить”

При нажатии `[Построить]` открывается список доступных построек.

Для каждой постройки нужно показывать:

```text
название
стоимость ресурсов
стоимость места
условия
эффект
```

Пример:

```text
Склад
Место: 5
Требует: 20 дерева
Эффект: +хранение ресурсов
```

Если построить нельзя, причина должна быть понятной:

```text
Недостаточно места
Нужно: 5
Свободно: 3
```

или:

```text
Нельзя строить на воде
```

---

## 16. Режим строительства

В режиме строительства карта должна помогать выбрать место.

Показывать:

- доступные гексы;
- недоступные гексы;
- вместимость;
- причины запрета;
- подходящие участки.

### Визуальные состояния

| Состояние | Визуал |
|---|---|
| можно строить | мягкая зелёная подсветка |
| можно, но не идеально | жёлтая подсветка |
| нельзя | красная/серая подсветка |
| выбранный участок | яркая рамка |
| не открыт | затемнение |

В режиме строительства не нужно показывать сложную логистику.

---

## 17. Режим обзора

Режим обзора — стандартный режим карты.

Показывает:

- местность;
- участки;
- основные постройки;
- выбранный гекс;
- простые дороги;
- базовые статусы.

Не показывает:

- все маршруты;
- все качества;
- все зоны влияния;
- все рабочие назначения.

---

## 18. Режим логистики

Режим логистики можно сделать в V1.1, но структуру стоит предусмотреть.

Показывать:

- дороги;
- тропы;
- подключение складов;
- маршруты доставки;
- проблемные связи;
- перегруженные участки позже.

В V1 можно ограничиться визуальным показом дорожных связей.

---

## 19. Отображение дорог

В логике дороги — это связи между соседними гексами.

В UI V1 нужно выбрать простой временный визуал.

Рекомендуемый вариант V1:

```text
дорога отображается как тонкий сегмент между соседними гексами, совпадающий с направлением движения
```

Требования:

- дорога должна визуально соединять два соседних участка;
- дорога не должна ломать арт гекса;
- юнит/ресурсный поток позже должен двигаться по той же линии;
- разные типы связи должны отличаться визуально.

### Типы связи V1

| Тип | Визуал |
|---|---|
| `trail` | тонкая земляная тропа |
| `road` | более широкая утрамбованная дорога |

Позже:

| Тип | Визуал |
|---|---|
| `rail` | рельсы |
| `power_line` | ЛЭП |
| `pipeline` | труба |
| `canal` | канал |

---

## 20. Не фиксировать финальный художественный стиль дорог в V1

Визуал дорог пока спорный.

Варианты, которые обсуждались:

1. дорога поверх гекса;
2. дорога по рёбрам;
3. дорожные маркеры;
4. гибрид;
5. технические швы между гексами;
6. центр-сегменты через гекс.

Для V1 нужно реализовать так, чтобы визуал дороги можно было заменить без изменения модели данных.

Главное для V1:

```text
RoadLink существует в данных.
UI умеет отрисовать связь.
```

---

## 21. Верхняя панель ресурсов

Верхняя панель должна показывать основные ресурсы.

Минимальный набор V1:

```text
дерево
камень
глина
еда
инструменты
население
```

Пример:

```text
Дерево: 120
Камень: 80
Глина: 40
Еда: 60
Инструменты: 5
Население: 8/12
```

Не нужно перегружать верхнюю панель будущими ресурсами вроде топлива, компонентов, науки и влияния, если они ещё не используются.

---

## 22. Нижняя панель действий

Нижняя панель может содержать быстрые действия:

```text
[Обзор]
[Строительство]
[Дороги]
[Задачи]
[Сохранить]
```

Или компактные кнопки:

```text
Построить
Проложить тропу
Улучшить участок
Снести
```

На раннем этапе допустимо обойтись кнопками в правой панели участка.

---

## 23. Левая панель режимов

Если делается левая панель, минимальный набор:

```text
Обзор
Строительство
Дороги
```

Позже:

```text
Логистика
Ресурсы
Территория
Рабочие
Проблемы
Экология
```

В V1 не обязательно реализовывать все режимы.

---

## 24. Статусы и предупреждения

На гексах и в панели можно показывать простые статусы:

```text
нет дороги
не хватает места
строится
нет ресурсов
нет рабочих
повреждено
```

Визуально:

- маленькая иконка;
- цветной маркер;
- текст в панели участка.

Не нужно выводить большие всплывающие сообщения на каждом гексе.

---

## 25. UI для строительства дороги

При выборе действия `Проложить тропу` или `Проложить дорогу`:

1. игрок выбирает стартовый гекс;
2. затем выбирает соседний гекс;
3. создаётся `RoadLink`;
4. связь отображается на карте;
5. в панели участка обновляется список подключений.

В V1 можно разрешить дороги только между соседними гексами.

Позже можно сделать автоматическую прокладку маршрута через несколько гексов.

---

## 26. UI для улучшения участка

В панели участка можно сделать блок:

```text
Улучшения участка:
[Расчистить]
[Выравнивание]
```

Пример:

```text
Расчистка
Эффект: +2 вместимости
Стоимость: 10 инструментов
```

Если улучшение недоступно:

```text
Недоступно: не хватает инструментов
```

---

## 27. UI для ресурсов участка

Если участок имеет ресурс, показывать:

```text
Ресурс: Глина
Запас: средний
Качество добычи: высокое
```

Или проще:

```text
Ресурс: Глина
Качество: высокое
```

На карте можно показывать маленькую иконку ресурса.

---

## 28. Качества участка в UI

Качества лучше показывать словами, а не только цифрами.

В данных:

```text
farmQuality: 80
buildQuality: 60
resourceQuality: 20
```

В UI:

```text
Сельское хозяйство: высокое
Строительство: среднее
Добыча: низкая
```

Цифры можно показывать позже в расширенном режиме.

---

## 29. Цветовая логика UI

Рекомендуемая цветовая логика:

| Смысл | Цвет |
|---|---|
| доступно | зелёный |
| предупреждение | жёлтый |
| ошибка / нельзя | красный |
| выбранный объект | голубой / синий |
| дорога | коричневый / песочный |
| тропа | светло-коричневый |
| вода | голубой |
| загрязнение позже | фиолетовый / грязно-серый |

Цвета не должны быть слишком кислотными.

---

## 30. Мобильность и маленькие экраны

Поскольку пользователь может смотреть проект и с мобильных устройств, UI не должен зависеть только от hover.

Важная информация должна открываться по клику/тапу.

Hover — только дополнительная подсказка.

---

## 31. Что не делать в UI V1

Не делать:

- мини-гексы внутри гекса;
- внутреннюю карту каждого участка;
- сложную анимацию всех рабочих;
- отображение всех маршрутов сразу;
- отображение всех будущих слоёв;
- большой политический слой;
- карту планеты;
- орбитальный интерфейс;
- свободное размещение зданий мышкой внутри гекса;
- сложные 3D-здания;
- перегруженную панель с десятками параметров.

---

## 32. Критерии готовности UI V1

UI локальной карты V1 считается готовым, если:

- карта отображает flat-top гексы;
- гекс можно выбрать;
- выбранный гекс подсвечивается;
- открывается панель выбранного участка;
- в панели видны местность, тип участка, вместимость и постройки;
- можно войти в режим строительства;
- доступные/недоступные участки отображаются понятно;
- можно построить базовую постройку;
- можно увидеть, почему строительство невозможно;
- можно создать тропу/дорогу между соседними гексами;
- дорога видна на карте;
- интерфейс не перегружен;
- все подписи на русском языке;
- карта остаётся читаемой.

---

## 33. Главный принцип для агента

```text
Не пытаться сразу сделать финальный красивый интерфейс.
Сначала сделать понятный рабочий UI локальной карты.
Все спорные визуальные решения оставить заменяемыми слоями.
```

Если агенту нужно изменить визуал дорог, гексов или панели, он должен не ломать модель данных и не менять базовые решения:

```text
flat-top hex
гекс = участок
вместимость
постройки внутри участка
дороги = RoadLink между соседними гексами
панель участка по клику
```
