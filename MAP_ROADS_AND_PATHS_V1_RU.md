# MAP_ROADS_AND_PATHS_V1_RU

## SpaceGame — дороги, тропы и натоптанные пути V1

Документ для ИИ-агентов и разработчиков.  
Цель — жёстко зафиксировать механику дорог и троп для первой версии локальной карты SpaceGame.

Документ дополняет:

- `MAP_SYSTEM_V1_RU.md`
- `MAP_DATA_MODEL_V1_RU.md`
- `MAP_UI_V1_RU.md`

---

## 1. Главный принцип

В SpaceGame нужно разделять две разные механики:

```text
1. Автоматическая натоптанность пути от повторяющегося движения.
2. Ручное строительство / улучшение дорог игроком.
```

Агент не должен смешивать эти механики в одну.

Правильная формула:

```text
Люди ходят → появляется след / тропа.
Игрок строит → появляется улучшенная тропа / дорога.
```

---

## 2. Что НЕ должен делать агент

Агенту запрещено самостоятельно:

- делать дороги случайно;
- создавать дороги без понятной связи между соседними гексами;
- автоматически улучшать путь до дороги без решения игрока;
- строить дороги не между соседними гексами;
- прокладывать маршруты без UI-контроля игрока;
- смешивать натоптанность и построенную дорогу в одно поле;
- делать свободное рисование дорог мышкой без тайлов;
- делать дороги поверх всех гексов без привязки к RoadLink;
- менять базовую модель гекса;
- менять flat-top ориентацию гексов;
- внедрять железные дороги, ЛЭП, трубопроводы в V1 без отдельного задания;
- делать сложную логистику до реализации базовой RoadLink-модели.

Если агент считает, что нужно отклониться от этого документа, он должен сначала описать предложение отдельно, но не внедрять самовольно.

---

## 3. Базовая сущность: RoadLink

Дорога, тропа или след — это не объект внутри одного гекса.

Это связь между двумя соседними гексами.

```text
RoadLink = связь Hex A ↔ Hex B
```

RoadLink существует только между соседними гексами, имеющими общую грань.

В V1 все RoadLink считаются двусторонними.

---

## 4. Обязательная структура RoadLink V1

Минимальная структура:

```text
RoadLink:
  id
  fromHexId
  toHexId
  direction

  footTraffic
  naturalPathLevel
  builtRoadLevel
  effectivePathLevel

  condition
  speedModifier
  capacity

  ownerId
  lastUsedAt
  status[]
```

---

## 5. Пояснение полей RoadLink

| Поле | Назначение |
|---|---|
| `id` | уникальный идентификатор связи |
| `fromHexId` | первый гекс |
| `toHexId` | соседний гекс |
| `direction` | направление от fromHexId к toHexId |
| `footTraffic` | накопленное количество проходов |
| `naturalPathLevel` | уровень естественной натоптанности |
| `builtRoadLevel` | уровень ручного улучшения игроком |
| `effectivePathLevel` | итоговый уровень пути |
| `condition` | состояние дороги/тропы |
| `speedModifier` | влияние на скорость перемещения |
| `capacity` | пропускная способность |
| `ownerId` | владелец построенной дороги, если есть |
| `lastUsedAt` | когда связь использовалась в последний раз |
| `status[]` | статусы связи |

---

## 6. Direction

Для flat-top hex использовать 6 направлений:

```text
E
NE
NW
W
SW
SE
```

Противоположные направления:

```text
E  ↔ W
NE ↔ SW
NW ↔ SE
```

Агент обязан использовать одну стабильную систему направлений во всём коде.

---

## 7. footTraffic

`footTraffic` — это накопленный след от фактического движения персонажей, рабочих, носильщиков, отрядов или транспорта.

Каждый раз, когда сущность проходит из одного гекса в соседний через RoadLink:

```text
footTraffic += 1
lastUsedAt = currentTime
```

Если RoadLink ещё не существует, но перемещение произошло, нужно создать RoadLink с начальными значениями:

```text
footTraffic = 1
naturalPathLevel = none
builtRoadLevel = none
```

---

## 8. naturalPathLevel

`naturalPathLevel` — уровень естественно натоптанного пути.

Он повышается автоматически от `footTraffic`.

Рекомендуемые уровни V1:

```text
none
trace
footpath
stable_footpath
```

Таблица:

| naturalPathLevel | footTraffic | Значение |
|---|---:|---|
| `none` | 0–9 | нет видимого следа |
| `trace` | 10–29 | слабый след |
| `footpath` | 30–79 | натоптанная тропа |
| `stable_footpath` | 80+ | устойчивая тропа |

В V1 можно использовать именно эти пороги.  
Если баланс будет неудобным, их можно вынести в настройки.

---

## 9. builtRoadLevel

`builtRoadLevel` — уровень ручного улучшения дороги игроком.

Он НЕ должен повышаться автоматически от `footTraffic`.

Рекомендуемые уровни V1:

```text
none
built_trail
dirt_road
```

Значение:

| builtRoadLevel | Кто создаёт | Смысл |
|---|---|---|
| `none` | никто | построенной дороги нет |
| `built_trail` | игрок | намеренно проложенная тропа |
| `dirt_road` | игрок | грунтовая дорога |

Позже можно добавить:

```text
paved_road
rail
power_line
pipeline
highway
```

Но не в V1.

---

## 10. effectivePathLevel

`effectivePathLevel` — итоговое качество связи.

Оно рассчитывается из:

```text
naturalPathLevel
builtRoadLevel
condition
```

Базовое правило V1:

```text
effectivePathLevel = лучший уровень из naturalPathLevel и builtRoadLevel
```

Пример:

```text
naturalPathLevel = footpath
builtRoadLevel = none
effectivePathLevel = footpath
```

```text
naturalPathLevel = trace
builtRoadLevel = dirt_road
effectivePathLevel = dirt_road
```

---

## 11. Порядок уровней качества пути

Рекомендуемый порядок от слабого к сильному:

```text
none
trace
footpath
stable_footpath
built_trail
dirt_road
```

Важно:

- `stable_footpath` может быть лучше обычной случайной тропы;
- `built_trail` — сознательно сделанная тропа, обычно надёжнее естественной;
- `dirt_road` — полноценная ранняя дорога.

---

## 12. speedModifier

`speedModifier` рассчитывается по `effectivePathLevel`.

Рекомендуемые значения V1:

| effectivePathLevel | speedModifier |
|---|---:|
| `none` | 1.00 |
| `trace` | 1.05 |
| `footpath` | 1.15 |
| `stable_footpath` | 1.25 |
| `built_trail` | 1.30 |
| `dirt_road` | 1.50 |

Это означает:

```text
1.00 = обычная скорость
1.50 = на 50% быстрее
```

Значения предварительные и могут балансироваться.

---

## 13. capacity

`capacity` — пропускная способность связи.

Для V1 можно использовать простую шкалу:

| effectivePathLevel | capacity |
|---|---:|
| `none` | 1 |
| `trace` | 1 |
| `footpath` | 2 |
| `stable_footpath` | 3 |
| `built_trail` | 3 |
| `dirt_road` | 5 |

В V1 capacity может быть подготовлена в данных, но не обязательно глубоко использоваться.

---

## 14. condition

`condition` — состояние пути.

В V1 можно использовать:

```text
normal
damaged
blocked
```

На старте достаточно всегда ставить:

```text
condition = normal
```

Позже состояние дороги будет нужно для:

- паводков;
- грязи;
- зимы;
- повреждений;
- атак;
- ремонта.

---

## 15. Автоматическое создание следов

Механика натоптанности должна работать так:

1. Рабочий/персонаж выбирает путь.
2. При переходе из гекса A в соседний гекс B игра находит RoadLink A-B.
3. Если RoadLink нет, он создаётся.
4. У RoadLink увеличивается `footTraffic`.
5. После изменения `footTraffic` пересчитывается `naturalPathLevel`.
6. После этого пересчитывается `effectivePathLevel`.
7. Визуал дороги/следа обновляется.

Важно:

```text
Автоматическая натоптанность НЕ тратит ресурсы.
Автоматическая натоптанность НЕ создаёт builtRoadLevel.
```

---

## 16. Ручное строительство дорог

Игрок должен иметь возможность осознанно улучшать путь.

Для этого нужен режим:

```text
Режим: Дороги
```

В этом режиме игрок может:

- выбрать связь между двумя соседними гексами;
- увидеть состояние связи;
- улучшить связь до built_trail;
- улучшить built_trail до dirt_road.

---

## 17. Выбор связи между гексами

В режиме дорог игрок должен выбирать именно связь, а не только гекс.

Минимально допустимый UX:

```text
1. Игрок кликает первый гекс.
2. Игрок кликает соседний гекс.
3. UI выбирает связь между ними.
4. Открывается панель выбранного RoadLink.
```

Альтернативно можно сделать кликабельные ребра/сегменты дороги, но это не обязательно для V1.

---

## 18. Панель выбранного RoadLink

При выборе связи показывать панель:

```text
Связь: Гекс A ↔ Гекс B
Тип местности A: ...
Тип местности B: ...

Натоптанность:
- Проходов: 42
- Естественный путь: Натоптанная тропа

Построенная дорога:
- Уровень: нет / построенная тропа / дорога

Итог:
- Скорость: +15%
- Пропускная способность: 2

Действия:
[Проложить тропу]
[Улучшить до дороги]
```

Если действие недоступно, причина должна быть видна:

```text
Недоступно: не хватает дерева
Недоступно: гексы не соседние
Недоступно: связь заблокирована
```

---

## 19. Стоимость ручных улучшений V1

Предварительные стоимости:

| Действие | Требования |
|---|---|
| Проложить тропу | 5 дерева или 5 труда |
| Улучшить до дороги | 15 дерева + 10 камня или эквивалент труда |

Если в текущей игре нет ресурса `труд`, использовать существующие ресурсы.

Если баланс проекта уже имеет свои ресурсы, агент должен адаптировать стоимость к текущим ресурсам, но не менять механику.

---

## 20. Улучшение одной связи

Минимальная операция:

```text
upgradeRoadLink(roadLinkId, targetLevel)
```

Правила:

- `targetLevel = built_trail` можно применять, если builtRoadLevel ниже;
- `targetLevel = dirt_road` можно применять, если есть built_trail или достаточные технологии;
- улучшение требует ресурсов;
- после улучшения пересчитать `effectivePathLevel`.

---

## 21. Улучшение маршрута из нескольких связей

В V1 можно НЕ делать.

Но архитектурно предусмотреть будущую операцию:

```text
upgradeRoute(roadLinkIds[], targetLevel)
```

Позже игрок сможет:

```text
выбрать несколько гексов → построить/улучшить путь по всей цепочке
```

В текущей задаче достаточно сделать улучшение одной связи.

---

## 22. Автоматические пути и ручные дороги должны сосуществовать

Пример:

```text
Связь A-B:
footTraffic = 65
naturalPathLevel = footpath
builtRoadLevel = none
effectivePathLevel = footpath
```

Игрок нажимает:

```text
[Улучшить до дороги]
```

После:

```text
footTraffic = 65
naturalPathLevel = footpath
builtRoadLevel = dirt_road
effectivePathLevel = dirt_road
```

`footTraffic` не обнуляется.

---

## 23. Деградация путей

В V1 деградацию можно не реализовывать.

Не нужно делать:

- зарастание троп;
- разрушение дорог;
- сезонные ухудшения;
- обслуживание дорог.

Но поля `condition` и `lastUsedAt` добавить можно, чтобы позже реализовать деградацию.

---

## 24. Визуальные уровни путей V1

В обычном режиме:

| Уровень | Визуал |
|---|---|
| `none` | ничего |
| `trace` | очень слабый пунктир / лёгкий след |
| `footpath` | тонкая тропа |
| `stable_footpath` | более заметная тропа |
| `built_trail` | чёткая тропа |
| `dirt_road` | более широкая земляная дорога |

Визуал не должен закрывать арт гекса.

---

## 25. Визуал в режиме дорог

В режиме дорог нужно показывать больше информации:

- все существующие RoadLink;
- возможные связи с соседями выбранного гекса;
- уровни путей;
- выбранную связь;
- доступные улучшения.

Цветовая логика:

| Состояние | Визуал |
|---|---|
| существующий путь | нейтральный цвет |
| выбранная связь | голубая/светлая подсветка |
| можно улучшить | зелёный акцент |
| нельзя улучшить | красный/серый акцент |
| автоматическая тропа | коричневый/песочный |
| построенная дорога | более чёткий коричнево-серый |

---

## 26. Совпадение визуала и движения

Визуал дороги должен совпадать с направлением движения персонажей/ресурсов.

Агент не должен делать так, чтобы:

```text
дорога визуально идёт сбоку,
а персонаж движется по центру гексов отдельно от дороги.
```

Если визуал дороги пока временный, он всё равно должен отражать связь между теми гексами, между которыми идёт движение.

---

## 27. Подключение объектов к дорогам

В V1 не нужно делать сложные логистические порты зданий.

Для V1 достаточно правила:

```text
Если гекс имеет RoadLink хотя бы с одним соседним гексом,
участок считается подключённым к дорожной сети.
```

Позже можно добавить:

- входы объектов;
- стороны подключения;
- подъезды к складам;
- погрузочные зоны.

---

## 28. Как работает сбор ресурсов

Если рабочий ходит собирать ресурс и возвращается:

1. игра строит путь по соседним гексам;
2. каждый переход увеличивает `footTraffic`;
3. часто используемые связи постепенно превращаются в следы и тропы;
4. скорость движения по этим связям повышается;
5. игрок видит, где реально часто ходят рабочие.

Это естественная автоматическая часть.

---

## 29. Как работает ручная дорога к руднику или складу

Если игрок хочет улучшить стратегически важное направление:

1. включает режим `Дороги`;
2. выбирает связь между соседними гексами;
3. смотрит текущий уровень пути;
4. нажимает `Проложить тропу` или `Улучшить до дороги`;
5. тратит ресурсы;
6. builtRoadLevel повышается;
7. скорость/пропускная способность улучшается.

Это ручная управляемая часть.

---

## 30. UI-режим “Дороги”

Минимальные элементы режима:

```text
Режим: Дороги
Выбранный гекс
Соседние связи
Панель выбранной связи
Кнопки улучшения
```

При выборе гекса подсвечивать 6 соседних направлений.

Если между гексами уже есть RoadLink — показывать его уровень.

Если RoadLink нет — показывать возможную связь:

```text
Можно проложить тропу
```

---

## 31. Ошибки, которые нельзя допускать

Агент не должен реализовывать поведение:

```text
Любое движение сразу создаёт построенную дорогу.
```

```text
Тропа появляется непонятно где, без связи с фактическими переходами.
```

```text
Игрок не может выбрать, где улучшать дорогу.
```

```text
Дорога строится на гексе, а не между двумя гексами.
```

```text
Нельзя узнать, почему путь появился.
```

```text
Нельзя увидеть количество проходов или уровень пути.
```

---

## 32. Минимальный алгоритм при перемещении

Псевдологика без кода:

```text
onMoveBetweenHexes(fromHexId, toHexId):
  убедиться, что гексы соседние
  найти RoadLink между ними
  если RoadLink нет:
    создать RoadLink
  roadLink.footTraffic += 1
  roadLink.lastUsedAt = now
  пересчитать naturalPathLevel
  пересчитать effectivePathLevel
  пересчитать speedModifier
```

---

## 33. Минимальный алгоритм улучшения

```text
upgradeRoadLink(roadLinkId, targetLevel):
  найти RoadLink
  проверить доступность targetLevel
  проверить ресурсы
  списать ресурсы
  roadLink.builtRoadLevel = targetLevel
  пересчитать effectivePathLevel
  пересчитать speedModifier
```

---

## 34. Критерии готовности V1 дорог

Система дорог V1 считается готовой, если:

- RoadLink существует как связь между соседними гексами;
- движение между соседними гексами увеличивает `footTraffic`;
- `naturalPathLevel` растёт от частоты проходов;
- `builtRoadLevel` не меняется автоматически;
- игрок может вручную выбрать связь между двумя соседними гексами;
- игрок может улучшить выбранную связь;
- в UI видны:
  - уровень натоптанности;
  - уровень построенной дороги;
  - итоговый эффект;
- визуал дороги/тропы меняется в зависимости от уровня;
- поведение понятно игроку;
- агент не строит дороги самовольно.

---

## 35. Что отложить на будущее

Не делать в V1:

- автоматический выбор длинного маршрута для строительства;
- строительство дороги через много гексов за один раз;
- мосты;
- тоннели;
- железные дороги;
- ЛЭП;
- трубопроводы;
- пробки;
- обслуживание дорог;
- деградацию;
- сезонную грязь;
- разные дороги для разных типов транспорта;
- логистические склады по маршрутам;
- торговые караваны по расписанию.

Эти системы можно добавить позже после стабильной RoadLink-модели.

---

## 36. Итоговая формула

```text
RoadLink = связь между соседними гексами.

footTraffic = сколько раз реально ходили.
naturalPathLevel = что натоптали автоматически.
builtRoadLevel = что построил игрок.
effectivePathLevel = итоговое качество пути.
```

Главное:

```text
Автоматическая натоптанность создаёт следы.
Игрок вручную строит дороги.
Эти механики связаны, но не являются одним и тем же.
```
